# AGENTS.md — 던전 오피스 작업 안내 (에이전트/다음 세션용)

이 파일은 새 세션이나 에이전트가 이 저장소에서 작업을 이어받을 때 가장 먼저 읽는 오리엔테이션이다. **무엇을 만들고 있고, 지금까지 무엇을 했으며, 어디에 무엇이 있고, 다음에 무엇을 할지**를 담는다.

---

## 1. 프로젝트 한 줄 요약

**던전 오피스 (Dungeon Office)** — 던전 레이드 회사들을 거래처로 둔 대장간의 백오피스를 운영하는 1인용 모바일 게임. 플레이어는 사장이고, 일꾼들에게 탭으로 지시해 납품기한 안에 장비를 만들어 납품한다. 주사위 없는 결정론적 타임 매니지먼트이며, 유료 선불(광고 없음)로 출시 지향.

## 2. 현재 상태 (2026-07 기준)

- **단계: 설계 문서 완성 + 완전 모듈화 (v2.0).** 코드는 현재 트리에 없다(§6 참조).
- **기본 브랜치 `main`이 진실의 원천.** 옛 작업 브랜치 `claude/dungeon-office-design-review-yhjxz1`는 `main`과 동일 커밋이라 사실상 중복.
- 저장소는 `docs/`의 **설계 문서 10종만** 담고 있다. 비즈니스 문서·플랜·프로토타입 코드는 의도적으로 제거됨(§6에서 복구 가능).

## 3. 문서 아키텍처 (편집 전 반드시 숙지)

### 문서 작성 규약
- **충돌 시 GDD가 이긴다.** 모듈끼리 어긋나면 GDD 원칙(§3)이 최종 심판. GDD와 어긋나면 GDD를 먼저 개정.
- **모듈화 최대화.** 모듈화할 수 있는 것은 모듈로. 새 규칙이 GDD에 쌓이기 시작하면 모듈로 뽑아라.
- **상호 참조는 모듈 이름으로.** 예: `mechanics §4`, `progression §2`, `GDD 원칙 5`, `balance §6`. 문서 **내부** 참조는 자기 섹션 번호(`§4.2` 등). 이 둘을 섞지 말 것.
- 각 모듈은 헤더에 `상위 문서: GDD v2.0(컨셉·UX). 규칙 근거: mechanics.md·progression.md` 형식의 계보를 둔다.
- 구체 명칭·숫자는 "설명용 예시"이며 확정은 해당 모듈의 몫. **구조적 확정값**(계열 상한, 일꾼 수 범위)만 예외.

## 4. 확정된 핵심 결정 (재론 금지 — 바꾸려면 명시적 근거 필요)

- **게임명**: 던전 오피스 (Dungeon Office). (가칭 '탭탭 포지'는 폐기.)
- **배포**: 무료 다운로드 + 거래처 1 체험 + 본편 언락 **비소모성 IAP(결제 1회)**. 광고 없음, 게임플레이·저장 완전 오프라인.
- **엔진**: Godot 4.4 + 타입드 GDScript, **20Hz 고정 틱 결정론**(정수 전용 상태).
- **콘텐츠 규모**: 거래처 8 · 라운드 41 · 아이템 41 · 일꾼 **시작 2·상한 6** · 시설 9종(변환3·합성3(1 예약)·제작3·강화1).
- **강화 단계 상한 +3.** 강화는 확률·실패 없는 결정적 공정.
- **콤보 삭제**(v1.9). 판정 = 납품한 카드 값의 합. (재도입 시 조건은 GDD 부록 참조.)
- **11개 설계 원칙(GDD §3)이 헌법.** 특히: 확률 판정 없음 / 비자발적 손실은 과열 하나 / 계열 상한제 / 방치 생산 없음 / 선불·광고 없음.

## 5. 지금까지의 작업 히스토리 (요약)

시간 순. 상세는 `git log`.

1. **v1.8 원본 업로드 → 리뷰.** 콤보 점수·골드 모호성, 일시정지, 리텐션 등 지적.
2. **v1.9 개정.** 콤보 삭제, 유료 선불 전환, 별 게이트 표준화(정산 골드 차등 + 누적 별 요건 상시), 일꾼 2~6 구조 확정.
3. **하위 문서 9종 작성**(플랫폼·아트·UI·아이템·스테이지·밸런스·사운드·수익·마케팅) — 전문 에이전트 병렬.
4. **게임명 '던전 오피스' 확정, 배포 형태(무료+언락 IAP) 확정.**
5. **마스터 플랜·QA 계획 작성**, 개발 로드맵(M0~M3, 약 9~10개월, 2027-04 출시 지향) 수립.
6. **M0 프로토타입 착수·검증.** Godot 프로젝트 골격 + 정수 결정론 시뮬을 Python 레퍼런스로 구현해 **결정론을 실제로 증명**(같은 시드→같은 해시, 스냅샷 왕복, 봇 완주). 일꾼 2명 라운드 가동률 58% 등 첫 정량 관측 확보. *(이후 §6에서 제거.)*
7. **9개 문서 5축 정합성 검수 + 4축 수정.** BLOCKER(플랫폼 IAP 미반영 등)·정합성 이슈 일괄 수정.
8. **프로젝트 단순화**(사용자 요청). 게임 설계 문서만 남기고 비즈니스 문서·플랜·M0 코드 삭제(§6).
9. **GDD 완전 모듈화 → v2.0.** GDD를 컨셉·UX만 남기고(426→164줄), 규칙을 `mechanics.md`·`progression.md`로 분리. 전 문서 상호 참조 갱신, dead link 0.
10. **`main`을 기본 브랜치로 승격**(사용자가 직접), 전 작업 병합.

## 7. 앞으로의 방향 (다음 작업 후보)

우선순위 순. 무엇을 할지는 사용자와 확인.

1. **재미 게이트 검증이 여전히 최우선.** 종이 위에서 답할 수 없는 질문 — "콤보 없이 일꾼 2명 구간의 스케줄링이 재미있는가"(mechanics §13, progression 부록의 검증 질문). 개발 재개 시 §6의 M0 코드로 그레이박스 플레이테스트부터. **여기서 재미가 안 나면 아트/콘텐츠 투자 전에 GDD를 고친다.**
2. **콘텐츠 문서 정밀화.** items·stages·balance는 v0.1 초안. 특히 **balance는 41라운드 중 38개 커트라인이 미확정**(헤드리스 봇 산출로 v0.2 예정). 봇을 GDScript로 이식해 CI 검증에 편성하는 것이 그 선행 작업.
3. **표현 문서 구체화.** ui-ux 와이어프레임·art 에셋·sound SFX는 방향은 확정, 실제 제작/좌표는 미착. 버티컬 슬라이스(거래처 1) 단위로 구체화.
4. **개발 착수(그린라이트 시).** Godot 4.4, 20Hz 결정론, `sim/`(엔진 비의존) vs `game/`(뷰) 분리. §6의 M0 코드가 시작점. 환경 제약 주의: 개발 환경에 Godot 미설치일 수 있고 프록시가 github.com 릴리스를 막을 수 있음(그때 Python 레퍼런스로 로직 검증 → GDScript 미러 패턴 사용).
5. **미작성 예정 문서**(과거 계획됐던 것): 로컬라이제이션 계획(한/영), 출시 체크리스트(등급분류·개인정보처리방침), 라이브 운영 로드맵. 필요 시점은 개발 단계에 맞춰.
