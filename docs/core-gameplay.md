## 1. 목적과 범위

이 문서는 MVP 한 라운드가 시작된 뒤 제한 시간이 끝날 때까지 게임이 어떻게 작동하는지를 정의한다. 플레이어가 내리는 명령, 명령의 성공 조건, 상태 변화와 손실 조건이 이 문서의 책임이다.

이 문서가 답해야 하는 질문은 다음과 같다.

- 플레이어는 어떤 대상을 어떤 명령으로 조작하는가?
- 일꾼과 시설은 언제 점유되고 언제 해제되는가?
- 재료가 장비와 강화석으로 바뀌는 동안 어떤 상태를 거치는가?
- 무엇이 과열로 소실되고, 무엇은 소실되지 않는가?
- 의뢰는 언제 활성화되고 철회되며 어떤 장비와 매칭되는가?
- +1 강화가 왜 결정론적인가?
- 라운드 종료 시 어떤 결과를 진행 시스템에 넘기는가?

## 2. 명시적 비범위

다음은 이 문서에서 결정하지 않는다.

- 아이템·기자재의 ID, 이름·분류·용도: `CNT-ITEM-001`
- 레시피 입력·출력 관계: `CNT-RECIPE-001`
- 라운드별 의뢰 구성과 학습 목표: `CNT-MVP-ROUND-001`
- 작업 시간, 인벤토리 용량, 요청 점수, 커트라인: `BAL-MVP-001`
- 별, 해금, 계약, 재도전 기록: `SYS-PROG-001`
- 골드 보상과 설비 구매: `SYS-ECON-001`
- 화면 배치, 탭 타깃, 애니메이션과 문구: `UX-MVP-001`, `UI-MVP-001`
- 고정 틱·명령·동시 사건 경계: `TECH-SIM-001`
- 실행 JSON과 저장 스키마: `TECH-DATA-001`
- 가격, 결제, 체험판 등 상업 모델

## 3. MVP 라운드 상태

라운드 상태는 다음 항목의 합이다.

| 상태 | 의미 |
|---|---|
| 남은 시간 | 0이 되면 즉시 라운드가 종료된다. |
| 일꾼 | `idle` 또는 특정 작업에 배정된 `working` 상태다. |
| 시설 | 빈 상태, 입력 대기, 작업 중, 산출물 대기 중 하나다. 용광로 산출물 대기에는 과열 유예가 추가된다. |
| 인벤토리 | 유한한 슬롯에 들어 있는 아이템 단위 목록이다. |
| 활성 의뢰 | 현재 화면에서 인내 시간이 흐르는 의뢰 카드다. |
| 대기 의뢰 | 공개 시점은 지났으나 활성 슬롯이 없어 기다리는 스크립트 의뢰다. |
| 총점 | 성공적으로 납품한 의뢰 카드 점수의 합이다. |
| 정지 여부 | 정지 중에는 라운드 상태가 변하지 않는다. |

모든 아이템은 한 개의 독립 단위다. 인벤토리 한 슬롯에는 아이템 한 개만 들어가며 스택은 없다.

## 4. 작업 공간의 역할 문법

구체적인 기자재 ID와 이름은 `CNT-ITEM-001`이 소유한다. 코어는 그 기자재가 구현할 수 있는 역할과 작동만 고정한다.

| 역할 | 작동 | 일꾼 | 과열 |
|---|---|:---:|:---:|
| 원자재 공급원 | 콘텐츠가 허용한 원자재를 무한히 제공한다. | 없음 | 없음 |
| 무인 변환 시설 | 하나의 유효 입력을 받으면 즉시 작업을 시작한다. | 없음 | 있음 |
| 작업자 처리 시설 | 완성된 입력과 `start` 명령을 받아 제작·합성·강화를 수행한다. | 1명 | 없음 |
| 납품 목적지 | 장비를 활성 의뢰와 매칭한다. | 없음 | 없음 |
| 파기 목적지 | 아이템을 플레이어의 명시적 선택으로 제거한다. | 없음 | 없음 |

인벤토리는 기자재가 아니라 공용 보관 상태다. MVP의 일꾼 수와 설치된 기자재 수는 라운드 중 변하지 않는다.

## 5. 추상 명령

UI의 구체적인 탭 표현과 무관하게 시뮬레이션은 아래 명령만 받는다.

| 명령 | 선행 조건 | 성공 결과 | 실패 결과 |
|---|---|---|---|
| `move(source, destination, item)` | 출발지에 아이템이 있고 목적지가 해당 아이템을 받을 수 있다. | 아이템 한 개를 즉시 이동한다. | 상태 변화 없음과 거절 사유를 반환한다. |
| `start(facility)` | 작업자 시설의 입력이 완성되어 있고 유휴 일꾼이 있다. | 가장 낮은 ID의 유휴 일꾼 한 명을 배정하고 작업을 시작한다. | 상태 변화 없음과 거절 사유를 반환한다. |
| `store(facility_output)` | 시설 산출물이 있고 빈 인벤토리 슬롯이 있다. | 산출물 한 개를 첫 빈 슬롯으로 옮긴다. | 상태 변화 없음과 `inventory_full`을 반환한다. |
| `deliver(item)` | 아이템이 장비이고 매칭 가능한 활성 의뢰가 있다. | 장비와 의뢰를 제거하고 점수를 더한다. | 장비를 유지하고 거절 사유를 반환한다. |
| `discard(item)` | 플레이어가 소유하거나 시설에 대기 중인 아이템이다. | 아이템 한 개를 즉시 제거한다. | 상태 변화 없음. |
| `pause` | 라운드가 진행 중이다. | 시간과 모든 타이머를 정지한다. | 상태 변화 없음. |
| `resume` | 라운드가 정지 중이다. | 시간 진행을 재개한다. | 상태 변화 없음. |

이동·수납·납품·파기는 게임 시간 비용과 일꾼 비용이 없다. 잘못된 조합은 목적지가 받지 않으므로 재료가 소실되지 않는다.

원자재 공급원은 예외적으로 보유 수량을 차감하지 않는다. 공급원에서 유효한 목적지로 이동이 성공할 때 허용된 원자재 인스턴스 한 개를 생성하며, 이동이 거절되면 인스턴스를 만들지 않는다.

## 6. 시설과 일꾼 규칙

### 6.1 입력과 작업 시작

- 시설은 `CNT-RECIPE-001`에 등록된 조합의 입력만 받는다.
- 여러 입력이 필요한 시설은 작업 시작 전까지 입력을 나누어 받을 수 있다.
- 작업 시작 전의 입력은 인벤토리로 다시 이동할 수 있다.
- 무인 변환 시설은 완전한 입력을 받는 즉시 작업을 시작하며 일꾼을 사용하지 않는다.
- 제작대·합성대·강화 모루는 입력이 완성된 뒤 `start` 명령을 받아야 작업을 시작한다.
- 작업 시작 시 모든 입력을 소비하고 해당 레시피와 종료 시점을 고정한다.

### 6.2 작업 중과 완료

- 작업 중인 시설은 추가 입력을 받지 않는다.
- 시작된 작업은 취소하거나 다른 작업으로 바꿀 수 없다.
- 일꾼을 쓰는 작업은 완료 즉시 일꾼을 해제한다. 산출물을 치우지 않아도 일꾼은 유휴 상태가 된다.
- 완료 산출물이 시설에 남아 있는 동안 그 시설은 다음 작업을 시작할 수 없다.
- 대기열과 자동 예약은 없다.

## 7. 과열과 손실

과열은 MVP에서 유일한 비자발적 아이템 손실이다.

1. 무인 변환 시설의 작업이 끝나면 산출물과 함께 과열 유예 타이머가 시작된다.
2. 유예 안에 산출물을 인벤토리나 유효한 다음 시설로 옮기면 과열이 종료된다.
3. 유예가 0이 될 때 산출물이 여전히 용광로에 있으면 그 산출물만 소실되고 용광로가 빈다.

다음 상황에서는 아이템이 소실되지 않는다.

- 다른 시설의 완료 산출물을 방치함
- 인벤토리가 가득 참
- 의뢰가 철회됨
- 유효하지 않은 목적지나 레시피를 선택함
- 작업 시작이 거절됨
- 강화 작업을 수행함

`discard`만은 플레이어가 선택한 자발적 파기이므로 위 원칙의 예외가 아니다.

## 8. 인벤토리와 선제 비축

- 원자재, 가공재, 강화석, 장비를 모두 보관할 수 있다.
- 빈 슬롯이 없으면 `store`는 실패하지만 시설 산출물은 그대로 남는다. 용광로 산출물의 과열 타이머는 계속 흐른다.
- 인벤토리 아이템은 유효한 시설 입력, 납품대 또는 쓰레기통으로 이동할 수 있다.
- 브리핑에서 예고된 의뢰를 위해 재료·강화석·장비를 미리 만드는 것이 정식 전략이다.
- 라운드가 종료되면 인벤토리와 모든 라운드 아이템은 제거되며 다음 라운드로 이월되지 않는다.

## 9. 의뢰 수명주기

### 9.1 공개와 활성화

- MVP의 모든 의뢰는 `CNT-MVP-ROUND-001`의 순서와 `BAL-MVP-001`의 공개 시점으로 완전히 스크립트된다.
- RNG와 요청 풀 추첨은 사용하지 않는다.
- 공개 시점에 빈 활성 슬롯이 있으면 의뢰가 즉시 활성화되고 인내 타이머가 시작된다.
- 슬롯이 없으면 의뢰는 대기 FIFO에 들어간다. 활성 슬롯이 비는 즉시 가장 먼저 대기한 의뢰가 활성화되며, 인내 타이머는 그때 시작된다.
- 대기 의뢰는 플레이어가 선택하거나 거절할 수 없다.

### 9.2 철회

- 활성 의뢰의 인내가 0이 되면 의뢰는 철회된다.
- 철회는 점수 감점, 골드 손실, 아이템 손실을 일으키지 않는다.
- 철회 직후 빈 슬롯에는 대기 의뢰가 활성화된다.

### 9.3 납품 매칭

장비는 다음 조건을 모두 만족하는 활성 의뢰에 납품할 수 있다.

- 장비 ID가 요청 장비 ID와 같다.
- 장비 강화 단계가 요구 단계 이상이다.

여러 의뢰가 매칭되면 카드 점수가 가장 높은 의뢰를 우선하고, 동점이면 먼저 활성화된 의뢰를 우선한다. +1 장비를 +0 의뢰에 납품할 수 있지만 카드에 적힌 점수 외 추가 보상은 없다. 매칭 의뢰가 없으면 납품은 거절되고 장비는 유지된다.

## 10. 결정론적 +1 강화

- MVP 장비의 강화 단계는 `0` 또는 `1`뿐이다.
- 강화 입력은 강화 단계 0 장비 한 개와 콘텐츠가 지정한 강화 재료 한 개다.
- 유휴 일꾼 한 명을 배정하고 정해진 작업 시간이 지나면 같은 장비 ID의 강화 단계 1 장비가 반드시 나온다.
- 성공률, 치명타, 품질 편차, 파괴, 단계 하락은 존재하지 않는다.
- +1 장비는 다시 강화할 수 없다.

같은 초기 상태에서 같은 시점에 같은 명령열을 적용하면 라운드 결과는 항상 같아야 한다. MVP에는 RNG 소비 지점이 없다.

## 11. 라운드 종료 계약

- 남은 시간이 0이 되면 새 명령을 받지 않고 즉시 라운드를 종료한다.
- 진행 중 작업과 대기 중 아이템은 점수로 바뀌지 않는다.
- 활성·대기 의뢰도 추가 평가 없이 제거된다.
- 코어는 `round_id`, 최종 점수, 납품 내역, 철회 수, 과열 소실 수를 `SYS-PROG-001`에 전달한다.
- 별, 골드, 해금, 계약 여부는 코어가 결정하지 않는다.
- 메뉴에서 재시작을 선택하면 결과 패킷을 만들지 않고 해당 라운드의 초기 상태로 돌아간다.

| `TECH-DATA-001` | 영속화할 결과·진행 의미 | 실행 데이터와 저장 스키마 |

## 13. 미결정 사항

| 질문 | MVP에서의 처리 | 결정 방법과 시점 |
|---|---|---|
| `BAL-MVP-001`의 MVP 일꾼 구성에서 대기열·취소 불가가 답답한가? | 규칙을 유지한다. | 전체 MVP 라운드 관찰 테스트 후 `QA-MVP-001`에서 판정한다. |
| 인벤토리 압박이 계획을 만들고 있는가? | 용량은 `BAL-MVP-001` 값으로 고정한다. | R4·R5의 최고 점유율과 파기 빈도를 측정한다. |
| 의뢰 대기 FIFO가 화면 밖 압박으로 느껴지는가? | UI가 다음 의뢰 수만 알릴 수 있다. | R3 테스트 후 UX 명세 개정 여부를 결정한다. |

## 14. 검증 방법

- 동일 초기 상태와 동일 명령 로그를 두 번 실행해 최종 상태 해시가 같아야 한다.
- 유효하지 않은 이동·제작·납품이 어떤 아이템도 잃게 하지 않아야 한다.
- 용광로 외 경로에서 비자발적 아이템 손실이 발생하지 않아야 한다.
- +1 강화 명령을 반복 재생하면 항상 같은 완료 시점과 산출물을 내야 한다.
- 정지 중 상태 해시가 변하지 않아야 한다.

## 15. 완료 기준

- 모든 MVP 상태와 명령의 선행 조건·성공·실패 결과가 정의되어 있다.
- `CNT-ITEM-001`과 `CNT-RECIPE-001`의 모든 항목을 이 규칙만으로 처리할 수 있다.
- `CNT-MVP-ROUND-001`의 R1~R5에 규칙 공백이 없다.
- 수치가 이 문서에 중복되지 않고 `BAL-MVP-001`로 위임되어 있다.
- 상업 모델, 화면 레이아웃, 저장 구현이 섞여 있지 않다.
- `TECH-SIM-001`, `TECH-DATA-001`, `QA-MVP-001`이 구현·테스트 가능한 입력/출력 계약을 얻는다.
