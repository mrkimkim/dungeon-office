## 0. M0의 질문 — 재미 게이트

M0의 산출물은 완성된 게임이 아니라 **"이 게임이 재미있는가"에 대한 근거 있는 답**이다. 통과하면 M1(아트·사운드 현금 집행), 불통과하면 GDD 개정. 판정은 아래 부록 C 항목의 관측으로 한다.

| 우선순위 | 부록 C 질문 | M0에서 답하는 방식 |
|:---:|------|------|
| **1** | **C-4 대기열 없음·취소 불가 + 2일꾼** | 2일꾼 라운드가 "답답함"인가 "빡빡한 저글링"인가 — 사람 플레이 관측 + 봇 가동률 |
| **1** | **C-3 콤보 없는 병렬 운영 자생력** | 대형 라인 + 소형 납품 병행이 자연 발생하는가, 대형 집중이 지배 전략인가 |
| 2 | C-2 회생 불가 라운드 체감 | 실패 확정 라운드를 기한까지 두는 것이 지루한가 |
| 2 | C-1 투기 생산 템포 | 인벤토리 용량만으로 긴장이 유지되는가 |
| 3 | C-5 강화 의뢰 성립 | 예고·인내·할증이 공정 깊이를 감당하는가 (r2로 맛보기) |

C-6(입장 요건 벽)·C-7(누적 별 게이트)은 라운드 간 경제·진행 층이라 M0 범위 밖 — M2에서 검증.

## 1. 두 겹의 계측 — 봇(정량) + 사람(정성)

### 1.1 봇 계측 (자동, 매 빌드)
플랫폼 §7.5의 `bot_player`(현재 `tools/ref/bot.py`, M1에 GDScript 이식)가 헤드리스로 라운드를 완주하며 **정량 지표의 하한**을 산출한다. 사람 관측 전에 데이터가 먼저 말한다.

측정 지표:
- **일꾼 가동률** = 1 − (유휴 일꾼·틱 / 전체 일꾼·틱). 낮으면 "일꾼이 놀고 있다"의 객관 신호.
- **납품 수 / 점수 / 별** — 봇 하한이 ★1 이상인가(플랫폼 §7.5 검증식: 권장 설비로 클리어 가능).
- **과열 손실 수** — 봇조차 과열을 못 피하면 유예가 너무 짧다.
- **유휴 스파이크** — 유휴 일꾼이 오래 지속되는 구간의 위치·길이(대기열 없음의 답답함 후보 지점).

### 1.2 M0 현재 봇 관측치 (Python 레퍼런스, `tools/ref/run.py`)

| 라운드 | 구성 | 점수/별 | 납품 | 일꾼 가동률 | 과열손실 |
|---|---|---|---|---|---|
| r1 | 2일꾼 · 화로2+제작대2 | 100 / ★★★ | 8 | **58.0%** | 0 |
| r2 | 3일꾼 · 강화 파이프라인 | 130 / ★★ | 10 | **22.8%** | 0 |

**해석(주의: 탐욕 봇 = 하한, 숙련 사람의 상한 아님)**:
- r1 58% — 2일꾼이 절반 넘게 바쁘다. 화로(불) 처리량이 병목이라 일꾼이 이따금 대기 → C-4의 "답답함" 후보. **관건: 사람이 이 대기를 '다음 수 계획'으로 채우는가, '할 게 없어 멍때리는가'.**
- r2 22.8% — 강화 파이프라인(마력로 480틱)이 길어 일꾼 유휴가 큼. 탐욕 봇이 깊은 라인을 잘 못 굴린다는 뜻일 뿐 아니라, C-3의 핵심 위험(대형 공정 중 납품 공백)을 그대로 보여줌. **관건: 사람이 대형 라인 도는 동안 소형 납품으로 리듬을 잇는가.**
- 두 라운드 모두 과열 손실 0 — 현재 유예(용광로 200틱·마력로 120틱)는 봇도 감당. 사람 기준으론 더 조여야 긴장이 생길 수 있음(C-1과 함께 튜닝).

> 이 수치는 **임시 데이터·탐욕 봇 기준의 출발점**이다. M0의 목적은 이 숫자를 사람 관측으로 해석하고 데이터를 조정하는 것이다.

## 2. 사람 플레이테스트 프로토콜

### 2.1 대상·규모 (M0)
- **본인 + 지인 3~5명.** 통계가 아니라 신호 탐지가 목적. 게임 경험 스펙트럼을 섞는다(타임매니지먼트 경험자 2 + 비경험자 2).
- 각자 r1(2일꾼)·r2(강화)를 각 3회 이상. 첫 판(학습)과 3판째(숙달)의 체감 차이를 본다.

### 2.2 관측 방식 — "말하는 플레이(think-aloud)"
플레이어에게 **생각을 소리 내어 말하게** 하고 화면+음성을 녹화. 관측자는 개입하지 않는다. 핵심은 **발화**다:
- "할 게 없네" / "뭘 해야 하지" → **C-4 답답함 신호**.
- "아 저거 먼저 했어야" / "지금 이거 돌리는 동안 저거" → **C-3 병렬 계획이 발생하는 중**.
- "아까워" (과열·철회 후) → 손실이 학습인지 분노인지 톤 확인(원칙 3·§9.1).
- 침묵 + 손 멈춤 지속 → 유휴 스파이크와 대조(봇 계측 구간과 겹치는가).

### 2.3 라운드 후 5문항 (각 1~5점 + 한 줄)
1. 마지막 30초, 심장이 뛰었는가? (마감의 드라마, GDD §1.3)
2. 일꾼이 놀아서 답답한 순간이 있었는가? 언제? (**C-4**)
3. 한 번에 여러 라인을 굴리는 재미가 있었는가? (**C-3**)
4. 실패/과열/철회가 억울했는가, 납득됐는가? (원칙 3)
5. "한 판 더" 하고 싶은가?

### 2.4 판정 기준 (재미 게이트, 마스터 플랜 §5.1)
- **통과**: 문항 3·5 평균 ≥ 4.0, 문항 2에서 "답답함"이 구조적 지속(막힘)이 아니라 순간적(리듬)일 것. think-aloud에서 병렬 계획 발화가 비경험자에게서도 관측될 것.
- **불통과 → GDD 개정 발의**: 후보 — 콤보 재도입(부록 B), 시작 일꾼 3명, 라운드 단축, 인벤토리 압박 상향, 화로 처리량 상향(가동률 개선). **아트 현금 집행 전에 여기서 멈추는 것이 목적이다.**

## 3. 결정론·회귀 (기술 QA, 매 빌드)

재미 판정과 별개로, 시뮬의 결정론은 매 빌드 자동 검증한다(플랫폼 §3·§7.4).

| 검사 | 도구 | 통과 기준 |
|---|---|---|
| 결정론 | `tools/ref/run.py` T1 | 같은 (round, attempt) → 같은 최종 해시(반복 일치) |
| 시드 변주 | run.py T2 | 다른 attempt → 다른 의뢰 전개 |
| 스냅샷 왕복 | run.py T3 | 중단 저장→재개 = 무중단 해시 일치 |
| 상한 정합 | run.py T6 | ★3 커트라인 ≤ 이론 상한 |
| **교차 구현** | `tools/verify_replay.gd` (Godot 헤드리스) | GDScript 재생 해시 == Python 골든 픽스처 |

- 현재 상태: Python 레퍼런스 T1~T6 **전부 통과**(이 환경에서 실행 확인). PCG32 GDScript 이식은 `tools/ref/gd_emu.py`로 400 draw 일치 확인.
- **남은 1건**: `verify_replay.gd`는 Godot 바이너리가 필요해 사장님 로컬에서 실행·확인해야 한다(이 환경엔 Godot 미설치). 통과하면 교차 구현 결정론이 닫힌다.
