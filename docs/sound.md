## 0. 전제 (GDD에서 도출)

- **사운드는 이 게임 손맛의 절반이다.** 모든 조작이 탭 하나뿐이므로(mechanics §3.1), 조작의 보상은 제스처가 아니라 피드백 — 일꾼의 망치질 소리와 진동이다.
- **소리는 정보다.** 플레이어의 눈은 세 개의 시계(용광로·시설 출구·의뢰 게이지, mechanics §8)를 오간다. 귀가 그중 하나 이상을 대신 감시할 수 있어야 한다. 특히 과열 — 게임 유일의 비자발적 손실원 — 은 **화면을 보지 않아도 귀(또는 진동)만으로 인지 가능**해야 한다.
- **환경 3종을 모두 1급 시민으로.** 스피커(소형, 저역 소실) / 이어폰 / 무음(햅틱 대체). 무음 플레이는 지원 대상이지 열화 모드가 아니다.
- 라운드 2~4분, 일꾼 2→6명, 시설 10종 이내, 거래처 8곳. 유료 선불 프리미엄, 1인 개발(외주·라이브러리 전제).

## 1. 사운드 디렉션 4원칙

1. **정보음은 고정하고, 확인음은 변주한다.** 과열 경고·완료 벨처럼 뜻을 학습해야 하는 소리는 음색·음정을 절대 바꾸지 않는다(조건반사가 목표). 탭·타격처럼 자주 나는 확인음은 라운드로빈·피치 변주로 피로를 막는다.
2. **처벌하는 소리는 없다.** 실패도 가볍게(GDD 원칙 10). 거친 버저·불협 스팅어 금지. 게임에서 유일하게 "나쁜" 소리는 과열 소실 단 하나이며, 그마저 짧고 만화적이다.
3. **결정론의 소리.** 강화는 도박이 아니다(mechanics §10). 드럼롤·룰렛·상승 지터 등 확률 연출의 문법을 금지한다. 확정의 쾌감은 단단한 원샷 타격으로 표현한다.
4. **긴장은 마감에서만.** 음악의 긴장 레이어는 오직 납품기한에 반응한다. 공포·위협의 정서는 쓰지 않는다 — 바빠지는 주방이지, 침몰하는 배가 아니다.

## 2. 음악 방향

**확정: "동화 판타지 워크숍 팝"** — 어쿠스틱 포크 기반 + 경쾌한 팝 리듬 + 소량의 켈틱 색채. 판타지 상단 가도의 낙천적인 장사꾼 정서와 대장간의 노동 리듬을 합친다. 오케스트라 웅장함·다크 판타지·칩튠 순수 8비트는 배제한다(귀엽고 폭넓게 사랑받는 방향, GDD §1.1).

- **핵심 악기 팔레트**: 마림바·피치카토 현·아코디언·틴 휘슬·통기타·가벼운 브라스 스타카토, 그리고 **모루(anvil) 타격을 퍼커션으로** — 대장간의 정체성이자 게임플레이 망치질 SFX와 자연스럽게 섞이는 장치.
- **템포**: 플레이 트랙 기준 104~116 BPM. 라운드 내 템포 변화는 없다(§3.2 — 긴장은 템포가 아니라 레이어로).

|---|---|
| Overcooked! 2 | 마감 압박의 명랑한 표현, 라운드 음악의 긴장 레이어링 |
| 모여봐요 동물의 숲 | 일상 루프의 편안함, 반복 청취를 견디는 절제된 밀도 |
| Stardew Valley | 따뜻한 포크 정서, 1인 제작 규모에서의 완성도 기준 |
| Moonlighter | "장사하는 판타지"의 낭만, 상점·정산 화면의 톤 |
| A Short Hike | 가벼운 어쿠스틱 팔레트, 스템 레이어의 자연스러운 출입 |

### 3.2 플레이 트랙의 적응형 설계 (확정)

각 플레이 곡은 4개 스템으로 납품받아 미들웨어에서 수직 레이어링한다.

| 레이어 | 내용 |
|---|---|
| L1 베이스 | 리듬 기반 + 코드. 브리핑에서도 단독 사용 |
| L2 멜로디 | 메인 멜로디. 평시의 얼굴 |
| L3 드라이브 | 퍼커션 강화 + 모루 퍼커션. "바빠진 대장간" |
| L4 긴장 | 현 오스티나토 + 초침 계열 퍼커션. 오직 마감에만 반응 |

| 상태 | 조건 (먼저 도달하는 쪽) | 구성 |
|---|---|---|
| 평시 | 라운드 개시 | L1+L2 |
| 가속 | 잔여 기한 50% 미만 **또는** 진행 중 작업 4개 이상 | +L3 |
| 임박 | 잔여 30초 | +L4 (전 레이어) |
| 최후 | 잔여 10초 | 전 레이어 + BGM 로우패스 스윕, 초당 1회 카운트다운 틱(SFX, P0) |
| 기한 도래 | 0초 | 음악 즉시 컷 → 출정 나팔(SFX) → [성공 시] 장착 매핑 "챙" 시퀀스(§4.2) → 판정 스팅어 |

- 상태 전환은 마디 경계에 스냅(임박→최후만 즉시). 상태는 내려가지 않는다 — 라운드는 마감을 향해서만 흐른다.
- 템포·조성은 절대 바꾸지 않는다. 긴장은 오직 밀도와 음색으로.
- 일시정지: 음악을 끄지 않고 로우패스 + −8dB. 정지-계획은 정당한 플레이(mechanics §11)이므로 소리도 "차분한 계획의 방"이 된다.

### 3.3 거래처 8곳 변주 전략 (비용 관점 확정)

**8곡 풀 제작을 기각하고, "플레이 3곡 + 거래처 징글 8종" 체제로 확정한다.**

- 근거 1 — 서사: 플레이의 무대는 언제나 **대장간**이다. 던전은 화면 밖 세계이므로, 플레이 BGM은 대장간의 것이어야 하고 던전 테마가 점유할 이유가 없다.
- 근거 2 — 비용: 풀 트랙 8곡×4스템은 1인 개발 예산(§7)의 배수를 먹는다. 3곡이면 가도의 진행감(초반→후반의 밀도 상승)은 충분히 표현된다.
- 거래처의 청각 정체성은 **브리핑 징글 8종**(각 6~10초)이 담당한다: 가도 테마의 공통 멜로디를 거래처 테마에 맞는 악기·리듬로 변주(예: 화산계 거래처 = 낮은 브라스와 팀파니, 설원계 = 글로켄슈필과 현). 같은 멜로디의 변주이므로 작곡 단가가 낮고, "같은 가도의 다른 회사"라는 세계관 통일성도 얻는다.
- 결과 화면·장부에서도 해당 거래처 징글을 재사용한다.

### 4.3 판정 스팅어 (별 0~3성)

- **0성 (공략 실패)**: 코믹한 "휘유~" 휘슬 하강 + 부드러운 퍼커션 "툭". 2초. 단조의 장송곡 금지 — 팀은 무사히 후퇴했고 "다시" 버튼이 가장 크다(progression §8). 소리가 끝나기 전에 재도전 UI가 이미 떠 있어야 한다.
- **1성 (신승)**: 단정한 2초 팡파르. "됐다, 아슬아슬했다."
- **2성 (안정 공략)**: 밝은 3초 팡파르 + 가벼운 박수.
- **3성 (완벽 공략 — 개선 행진)**: 브라스 행진곡 스팅어 4~6초 + 군중 환호 + **모루 1타로 마침표**. 게임에서 가장 호화로운 소리 — "내 솜씨가 남의 개선 행진을 만들었다"(GDD 원칙 10)는 무게를 여기에 싣는다.
- 4종 모두 공통 모티프(가도 테마)의 변주로 작곡해 판정 간 서열이 귀로 즉시 비교되게 한다.

## 5. 믹싱·우선순위 정책

### 5.1 버스 구조와 덕킹

```
Master ─ BGM / SFX-P0(정보) / SFX-P1(확인) / SFX-P2(질감) / UI
```

- **덕킹**: P0 발화 시 BGM −6dB, P2 −9dB (어택 10ms / 릴리즈 400ms). P0는 어떤 상황에서도 뚫고 나온다.
- **주파수 슬롯 예약**: 완료 벨·과열 경고의 기본 에너지는 2~4kHz. 작곡 발주서에 "플레이 트랙은 2~4kHz 지속음을 피할 것"을 명기해 덕킹 없이도 정보음이 비집고 나올 자리를 확보한다.
- 소형 스피커 대응: P0/P1의 에너지는 500Hz~5kHz에 배치(모바일 스피커에서 저역은 소실된다). 저역의 웅장함은 이어폰용 보너스로만.

### 5.2 동시 다발 채널 관리 (최대 6일꾼 + 용광로들)

- **작업 루프는 시설 계열별 1보이스 원칙.** 망치질 3개가 돌아도 루프 인스턴스는 1개 — 동시 작업 수는 "밀도 파라미터"(타격 빈도·겹침·볼륨 +1.5dB/개, 상한 +4.5dB)로 표현한다. 6일꾼 풀가동은 소리의 아수라장이 아니라 "풍성하게 바쁜 대장간"이어야 한다.
- 과열 경고 다중 발생 시: 경고 보이스는 1개, 최고 단계가 우선한다. (어느 용광로인지의 식별은 화면 몫 — 소리는 "용광로 어딘가가 위험하다"까지만 책임진다.)
- 전체 보이스 상한 24. P2 루프 최대 8. 동일 SFX 50ms 내 재발화 금지(탭 스팸 대비).

### 5.3 반복 청취 피로 방지

- 타격·탭류(P1, 제작 루프 타격): **라운드로빈 4샘플 + 피치 ±40센트 랜덤.**
- 정보음(P0)·판정 스팅어: **변주 절대 금지.** 원칙 1이 우선한다.
- 작업 루프(P2)는 30초 이상 들어도 거슬리지 않도록 어택 성분을 부드럽게 — 긴장은 음악(L3/L4)의 일이지 루프음의 일이 아니다.

## 6. 햅틱 설계

### 6.1 원칙

- 햅틱은 장식이 아니라 **무음 플레이의 정보 채널**이다. P0 정보음 전원에 햅틱을 짝지운다.
- 일상 탭에는 햅틱을 넣지 않는다(2~4분 내내 울리는 진동은 피로다). 진동이 오면 "볼 일이 생겼다"는 뜻이어야 한다.
- 문법은 Android 최소공배수에 맞춘다: **강도 2단(약/강) × 패턴 4형(단발/2연/3연/장)** 만으로 성립하게 설계.

### 6.2 매핑

| 이벤트 | 패턴 | 평시 | 무음 부스트 시 |
|---|---|---|---|
| 작업 완료 벨 | 약·단발 | ○ | 강으로 승격 |
| 과열 경고 1단계 | 약·2연 | ○ | 강·2연 |
| 과열 경고 2단계 | 강·3연 (1초 주기 반복) | ○ | ○ |
| 과열 소실 | 장(300ms 저강도 럼블) | ○ | ○ |
| 납품 | 약·단발 | ○ | ○ |
| 강화 성공 | 강·단발 ("쿵" — 손에 남는 확정) | ○ | ○ |
| 착수 거절 | 약·2연 (짧게) | ○ | ○ |
| 마감 카운트다운 | 약·단발 ×10 (초당) | ○ | ○ |
| 별 3성 / 계약 체결 | 3연 상승 패턴 | ○ | ○ |
| 의뢰 도착 / 탭 확인 | 없음 | — | 의뢰 도착만 약·단발 |

- **무음 부스트**: SFX 볼륨 0 또는 기기 무음 감지 시 위 표의 승격 프로파일을 적용하는 토글(기본 켬). iOS의 무음 스위치 감지는 우회적이므로 "볼륨 0" 기준을 주 트리거로 한다.

### 6.3 플랫폼 노트

- **iOS**: Core Haptics로 강도·샤프니스 제어(주 타깃). 구형은 UIFeedbackGenerator 폴백.
- **Android**: VibrationEffect(진폭 제어는 API 26+, 컴포지션은 30+, 기기 편차 큼). §6.1의 단순 문법 덕에 waveform 폴백으로도 의미가 보존된다.
- 검증 기준: **눈 감고 소리 끄고**, 진동만으로 "완료 / 과열 위험 / 과열 임박"의 3상태를 구분할 수 있으면 합격.

## 7. 제작 전략과 견적

### 7.1 제작 구분 (확정)

| 자산 | 방식 | 근거 |
|---|---|---|
| 음악 전곡 + 스팅어 + 징글 | **외주 (작곡가 1인에게 일괄)** | 모티프 통일성(가도 테마의 변주 체계)이 생명 — 분할 발주 금지 |
| 핵심 SFX 15종 (망치질 세트, 완료 벨, 과열 3종, 강화, 납품, 판정 보조음 등) | **외주 사운드 디자인** | 손맛의 절반 — 라이브러리 짜깁기로 타협 불가한 부분 |
| 나머지 SFX (~40종) | **라이브러리 구매 + 자체 가공** (에디터에서 컷·피치·레이어링) | UI·앰비언스류는 기성 소스로 충분 |
| 햅틱 패턴 | 자체 제작 (엔진 내 데이터) | 코드 작업에 가까움 |
| 믹싱·오디오 버스 세팅 | 자체 + 외주 검수 1회 | 버스·덕킹 구성은 반복 조정이 잦아 내재화 (Godot 내장, §8) |

### 7.2 수량과 예산

- **음악**: 풀 트랙 6곡 + 스팅어 5종 + 거래처 징글 8종 (§3.1). 플레이 3곡은 4스템 분리 납품.
- **SFX**: 이벤트 약 35종, 라운드로빈·변주 포함 소스 **약 60~80개**.
- **예산 (개략, KRW)**: 음악 외주 300~550만 (곡당 40~70만 + 스팅어·징글 패키지) / 핵심 SFX 외주 60~120만 / 라이브러리 20~40만 / 예비·이터레이션 10% → **총 400~800만 원, 목표 550만 원.** 유료 선불 게임의 첫인상에서 오디오가 차지하는 비중을 감안하면 여기서 아끼지 않는다.
- 발주 순서: ① 가도 테마(메인 모티프) 1곡 확정 → ② 플레이 1곡 + 핵심 SFX로 수직 슬라이스 검증 → ③ 나머지 일괄. 모티프 확정 전 다곡 발주 금지.

## 8. 기술 사양

- **미들웨어: Godot 내장 오디오를 기본으로 확정** (플랫폼 문서의 Godot 채택과 동기화 — v0.1 개정). 적응형 레이어(§3.2)는 오디오 버스 + 스템별 AudioStreamPlayer 볼륨 제어로, 덕킹(§5.1)은 버스 이펙트(컴프레서 사이드체인 또는 코드 덕킹)로, 보이스 관리(§5.2)는 폴리포니 제한 + 자체 우선순위 큐로 구현한다 — 본 문서의 요구 수준(4스템·P0 덕킹·라운드로빈)은 내장 기능으로 충분하다. FMOD Studio는 기각 — Godot 연동이 비공식 GDExtension이라 모바일 익스포트의 유지보수 리스크가 1인 개발에 과하다. 단, 외주 사운드 디자이너가 FMOD 납품을 강하게 원할 경우에만 재검토(그 경우 GDExtension 검증을 수직 슬라이스 단계에서 완료할 것).
- **포맷**: 납품 원본 48kHz/24bit WAV. 인게임 — 음악은 스테레오 Vorbis 스트리밍(≈128~160kbps), SFX는 **모노** 44.1kHz Vorbis, 단 P0 정보음과 탭 확인음은 비압축 PCM 메모리 상주(지연 최소화 — 탭 응답이 손맛이다).
- **용량 예산**: 오디오 총 **40MB 이내** (음악 ≤25MB, SFX ≤10MB, 예비 5MB).
- **라우드니스**: BGM 통합 −18 LUFS 기준, P0 정보음 피크 −6dBFS. 스피커·이어폰 양쪽에서 QA.
- **오디오 옵션 (설정 화면)**:
  - BGM / SFX 볼륨 분리 슬라이더 (SFX만 켠 플레이를 정식 지원 — 정보음 중심 설계 덕에 성립)
  - 햅틱: 끔 / 약 / 표준
  - 무음 부스트 토글 (§6.2, 기본 켬)
  - 모노 다운믹스 토글 (한쪽 이어버드 플레이 대응)

## 9. 미결 사항 — 검증으로 답할 질문

1. **과열 경고의 관통력** — 6일꾼 풀가동 + 임박 레이어(L4) 위에서 경고 2단계가 실기기 스피커로 확실히 들리는가 (수직 슬라이스에서 최우선 검증)
2. **징글 8종의 식별력** — 같은 멜로디의 악기 변주만으로 플레이어가 거래처를 귀로 구분하는가, 아니면 배경 장식에 그치는가 (그쳐도 실패는 아님 — 식별의 주 채널은 화면)
3. **망치질-게이지 동기의 구현 비용** — 진행 게이지와 타격 리듬의 동기(§4.2)를 Godot 내장 오디오(재생 위치 폴링 + 시뮬 틱 연동)로 깔끔하게 구현할 수 있는가
4. **카운트다운 틱의 피로** — 매 라운드 듣는 10초 틱이 수십 회 재도전에서 짜증이 되는가 (대안: 잔여 5초로 축소)
5. **햅틱 3상태 구분** — §6.3의 검증 기준이 중저가 Android 실기에서 성립하는가
