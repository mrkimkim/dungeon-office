## 1. 목적과 범위

이 문서는 MVP에서 실제로 플레이할 거래처 한 곳과 라운드 다섯 개를 정의한다. 다섯 라운드는 새 규칙을 계속 추가하는 콘텐츠가 아니라 다음 핵심 가설을 차례로 검증하는 수직 슬라이스다.

```text
R1 기본 제작
-> R2 병렬 작업과 수납
-> R3 과열 관리와 의뢰 트리아지
-> R4 브리핑 예고와 선제 비축
-> R5 강화석 준비와 결정론적 +1 강화
```

## 2. 명시적 비범위

- 라운드 규칙 자체: `SYS-CORE-001`
- 아이템·레시피 정의: `CNT-ITEM-001`, `CNT-RECIPE-001`
- 모든 시간·점수·커트라인: `BAL-MVP-001`
- 튜토리얼 말풍선, 화면 플로우, 연출: `UX-MVP-001`, `UI-MVP-001`
- 상세 세계관, 캐릭터 대사, 최종 카피
- 추가 거래처·추가 라운드·추가 강화 단계·조립 콘텐츠
- 상업 모델

## 3. 거래처

| 필드 | 값 |
|---|---|
| 거래처 ID | `CLIENT_MVP_01` |
| 작업명 | 새싹 원정대 |
| 역할 | 첫 거래처이자 MVP 전체 학습의 의뢰인 |
| 라운드 | `R1`~`R5` |
| 완료 조건 | R5 별 1개 이상으로 계약 체결 |

작업명은 콘텐츠 식별을 위한 최소 이름이다. 성격, 외형, 문체와 로고는 세계관·아트 문서가 소유한다.

## 4. 시설 구성

### 4.1 R1~R4 기본 구성

- `FAC_SUPPLY`
- `FAC_FURNACE`
- `FAC_WEAPON_BENCH`
- `FAC_DELIVERY`
- `FAC_TRASH`
- 공용 인벤토리

### 4.2 R5 추가 구성

영구 상태 `enhancement_capability_owned`가 참이면 다음 시설을 함께 사용할 수 있다.

- `FAC_SYNTH_BENCH`
- `FAC_ENHANCE_ANVIL`

R5 요청은 이 두 시설 없이는 이행할 수 없으므로 R5 입장 조건과 정확히 일치한다.

## 5. 의뢰 템플릿

템플릿은 요청 대상과 콘텐츠 역할만 소유한다. 점수와 인내는 `BAL-MVP-001`이 소유한다.

| ID | 요청 장비 | 요구 단계 | 예고 | 역할 |
|---|---|:---:|:---:|---|
| `REQ_DAGGER_STD` | `EQ_DAGGER` | 0 | 아니오 | 기본 저가 의뢰 |
| `REQ_SWORD_STD` | `EQ_IRON_SWORD` | 0 | 아니오 | 두 주괴가 필요한 중가 의뢰 |
| `REQ_DAGGER_RUSH` | `EQ_DAGGER` | 0 | 아니오 | R3의 짧은 인내 의뢰 |
| `REQ_SWORD_RUSH` | `EQ_IRON_SWORD` | 0 | 아니오 | R3에서 우선순위를 고민하게 하는 의뢰 |
| `REQ_SWORD_FORECAST` | `EQ_IRON_SWORD` | 0 | 예 | R4 종반의 고가 예고 의뢰 |
| `REQ_DAGGER_P1` | `EQ_DAGGER` | 1 | 강화 입문 고지 | R5 첫 강화 실습 |
| `REQ_SWORD_P1_FORECAST` | `EQ_IRON_SWORD` | 1 | 예 | R5 종반의 최종 예고 의뢰 |

## 6. 스크립트 표기

모든 의뢰는 RNG 없이 절대 공개 시점으로 스크립트된다. 아래 순서 번호는 콘텐츠 순서의 Source of Truth이고, 각 이벤트의 `release_at`, 인내와 활성 슬롯 수는 `BAL-MVP-001`이 같은 순서 번호에 결합한다.

```text
E1 -> E2 -> ...
```

같은 공개 시각을 가진 이벤트는 같은 시각에 공개하며, 슬롯이 없을 때의 FIFO 대기는 `SYS-CORE-001`을 따른다.

## 7. R1 — 첫 납품

### 7.1 목적

공급함 → 용광로 → 무기 제작대 → 납품대의 한 사이클을 이해시키고, 첫 납품 직후 점수가 오르는 인과를 검증한다.

### 7.2 콘텐츠

- 사용 아이템: 철광석, 철 주괴, 단검
- 사용 레시피: `RCP_SMELT_IRON`, `RCP_CRAFT_DAGGER`
- 활성 의뢰 슬롯: `BAL-MVP-001`
- 스크립트: `E1 Dagger -> E2 Dagger -> E3 Dagger`

| 이벤트 | 템플릿 |
|---|---|
| `R1-E1` | `REQ_DAGGER_STD` |
| `R1-E2` | `REQ_DAGGER_STD` |
| `R1-E3` | `REQ_DAGGER_STD` |

### 7.3 콘텐츠 완료 조건

- 단검 한 건 납품만으로 라운드를 클리어할 수 있다.
- 과열 경고는 노출할 수 있지만 첫 강제 학습 항목으로 취급하지 않는다.
- 철검, 합성, 강화 요청은 없다.

## 8. R2 — 두 손의 리듬

### 8.1 목적

용광로가 무인으로 도는 동안 일꾼이 제작하고, 산출물을 인벤토리에 수납해 시설을 비우는 병렬 리듬을 검증한다.

### 8.2 콘텐츠

- 신규 아이템: 철검
- 신규 레시피: `RCP_CRAFT_IRON_SWORD`
- 스크립트: 단검과 철검을 번갈아 총 여섯 건

| 이벤트 | 템플릿 |
|---|---|
| `R2-E1` | `REQ_DAGGER_STD` |
| `R2-E2` | `REQ_SWORD_STD` |
| `R2-E3` | `REQ_DAGGER_STD` |
| `R2-E4` | `REQ_SWORD_STD` |
| `R2-E5` | `REQ_DAGGER_STD` |
| `R2-E6` | `REQ_SWORD_STD` |

### 8.3 콘텐츠 완료 조건

- 첫 두 의뢰는 동시에 공개되어 병렬 계획의 필요를 보여준다.
- 전체 요청은 권장 구성에서 전부 이행 가능하다.
- 수납을 쓰지 않아도 클리어는 가능하지만, 연속 생산에는 수납이 명백히 유리해야 한다.

## 9. R3 — 바쁜 주문대

### 9.1 목적

용광로 과열을 실제 손실원으로 학습하고, 모든 의뢰를 처리할 수 없는 상황에서 철검과 단검 중 무엇을 우선할지 선택하게 한다.

### 9.2 콘텐츠

- 신규 의뢰: `REQ_DAGGER_RUSH`, `REQ_SWORD_RUSH`
- 스크립트: 단검과 철검을 번갈아 일곱 쌍, 총 열네 건

| 이벤트 | 템플릿 | 이벤트 | 템플릿 |
|---|---|---|---|
| `R3-E1` | `REQ_DAGGER_RUSH` | `R3-E2` | `REQ_SWORD_RUSH` |
| `R3-E3` | `REQ_DAGGER_RUSH` | `R3-E4` | `REQ_SWORD_RUSH` |
| `R3-E5` | `REQ_DAGGER_RUSH` | `R3-E6` | `REQ_SWORD_RUSH` |
| `R3-E7` | `REQ_DAGGER_RUSH` | `R3-E8` | `REQ_SWORD_RUSH` |
| `R3-E9` | `REQ_DAGGER_RUSH` | `R3-E10` | `REQ_SWORD_RUSH` |
| `R3-E11` | `REQ_DAGGER_RUSH` | `R3-E12` | `REQ_SWORD_RUSH` |
| `R3-E13` | `REQ_DAGGER_RUSH` | `R3-E14` | `REQ_SWORD_RUSH` |

### 9.3 콘텐츠 완료 조건

- 권장 구성의 이론적 철 주괴 공급량보다 전체 요청의 철 주괴 수요가 커야 한다.
- 따라서 전량 납품은 불가능하고 철회가 정상 플레이의 일부가 된다.
- 과열 소실 한 번을 겪어도 좋은 우선순위 판단으로 별 3개가 가능해야 한다.
- 철회에는 감점이 없다는 사실을 결과에서 확인할 수 있어야 한다.

## 10. R4 — 미리 준비하기

### 10.1 목적

브리핑에서 본 종반 철검 의뢰를 기억하고, 도착 전에 철검을 제작해 인벤토리에 보관하는 예고 → 비축 루프를 검증한다.

### 10.2 브리핑 의무 정보

- 종반에 `REQ_SWORD_FORECAST`가 온다.
- 요청 장비는 철검 +0이다.
- 일반 철검보다 높은 점수다.
- 도착 뒤 처음부터 만들기에는 시간이 매우 빠듯하므로 선제 제작이 권장된다.

### 10.3 콘텐츠

| 이벤트 | 템플릿 |
|---|---|
| `R4-E1` | `REQ_DAGGER_STD` |
| `R4-E2` | `REQ_SWORD_STD` |
| `R4-E3` | `REQ_DAGGER_STD` |
| `R4-E4` | `REQ_SWORD_STD` |
| `R4-E5` | `REQ_DAGGER_STD` |
| `R4-E6` | `REQ_SWORD_STD` |
| `R4-E7` | `REQ_DAGGER_STD` |
| `R4-E8` | `REQ_SWORD_FORECAST` |

### 10.4 콘텐츠 완료 조건

- 예고 의뢰를 제외한 점수만으로 별 2개까지 가능해야 한다.
- 별 3개에는 예고 의뢰 납품이 필요해야 한다.
- 예고 의뢰 공개 뒤 원자재부터 제작하면 기한 내 완료가 보장되지 않아야 한다.
- 미리 완성한 철검은 도착 즉시 납품할 수 있어야 한다.

## 11. R5 — 첫 강화 계약

### 11.1 목적

강화석 라인과 장비 라인을 병렬로 준비하고, +1 강화가 확률 없이 정확히 완료된다는 사실을 검증한다. R4의 예고·비축 학습을 더 깊은 공정에 적용하는 최종 라운드다.

### 11.2 입장 조건

- R4 클리어
- `enhancement_capability_owned = true`

### 11.3 브리핑 의무 정보

- 강화는 장비 +0과 강화석을 강화 모루에 넣어 수행한다.
- 실패·파괴·품질 편차가 없다.
- 중반에 +1 단검, 종반에 고가 `REQ_SWORD_P1_FORECAST`가 온다.
- 종반 의뢰를 위해 철검과 강화석을 미리 준비하는 것이 권장된다.

### 11.4 콘텐츠

| 이벤트 | 템플릿 |
|---|---|
| `R5-E1` | `REQ_DAGGER_STD` |
| `R5-E2` | `REQ_SWORD_STD` |
| `R5-E3` | `REQ_DAGGER_STD` |
| `R5-E4` | `REQ_DAGGER_P1` |
| `R5-E5` | `REQ_SWORD_STD` |
| `R5-E6` | `REQ_DAGGER_STD` |
| `R5-E7` | `REQ_SWORD_STD` |
| `R5-E8` | `REQ_SWORD_P1_FORECAST` |

### 11.5 콘텐츠 완료 조건

- +1 단검은 공개 뒤 처음부터 만들어도 인내 안에 완료할 수 있어야 한다.
- 종반 +1 철검은 공개 뒤 처음부터 만들 수 없고, 철검·강화석 비축으로 해결되어야 한다.
- 최종 예고 의뢰를 놓쳐도 라운드 클리어와 별 2개는 가능해야 한다.
- 별 3개에는 최종 예고 의뢰가 필요해야 한다.
- R5 별 1개 이상으로 `CLIENT_MVP_01` 계약이 체결된다.

## 12. 난이도·교육 불변 조건

1. 새 개념은 앞 라운드의 개념을 대체하지 않고 누적한다.
2. 가르치지 않은 레시피를 의뢰가 먼저 요구하지 않는다.
3. R3의 과부하는 전량 처리 실패를 의도한 콘텐츠이지 숨은 벌점이 아니다.
4. R4·R5 대형 의뢰는 브리핑에서 장비·강화 단계·도착 국면을 고지한다.
5. R5까지 모든 의뢰는 현재 시설과 레시피로 이행 가능하다.
6. 어떤 라운드도 새 입력 제스처나 별도 판정 축을 만들지 않는다.

## 14. 미결정 사항

| 질문 | MVP 처리 | 결정 방법 |
|---|---|---|
| R3 열네 건이 혼란이 아니라 선택으로 읽히는가? | 현재 순서를 유지한다. | 첫 플레이 철회 패턴과 인터뷰 확인 |
| R4 예고 한 번으로 선제 제작을 이해하는가? | 브리핑과 동일 표식을 실제 카드에 사용한다. | R4 무힌트 완료율 측정 |
| R5에 합성대와 강화 모루 두 시설을 함께 소개해도 되는가? | 강화 설비 세트로 묶고 한 라운드에서 가이드한다. | 첫 +1 단검 완료율 측정 |

## 15. 검증 방법

- 각 라운드의 모든 요청을 생산 그래프에서 역추적해 필요한 시설이 보유 집합에 포함되는지 검사한다.
- 스크립트에 RNG 풀이나 카탈로그 밖 요청이 없어야 한다.
- R3 총 철 수요가 이론 공급량보다 큰지 계산한다.
- R4·R5는 예고 의뢰 제외 시 별 2개 가능, 별 3개 불가능인지 계산한다.
- R5 최소 클리어 경로에 +1 장비 납품을 강제하지 않되, 라운드 안에서 +1 학습 기회가 반드시 활성화되는지 확인한다.

## 16. 완료 기준

- 거래처 한 곳과 R1~R5 전 콘텐츠가 정의되어 있다.
- 각 라운드가 하나의 주 학습·검증 목적을 가진다.
- 모든 의뢰가 아이템·레시피·시설과 참조 무결성을 가진다.
- 숫자와 화면 표현을 소유하지 않는다.
