## 1. 목적과 범위

이 문서는 MVP 라운드에서 생성·이동·보관·소비·납품할 수 있는 아이템과 실제 작업 공간에 존재하는 시설·일꾼 엔터티의 전 목록과 정체성을 정의한다. 목록에 없는 아이템과 기자재는 MVP 데이터와 화면에 존재할 수 없다.

## 2. 명시적 비범위

- 입력에서 출력으로의 변환 관계: `CNT-RECIPE-001`
- 작업 시간, 점수, 인내, 경제 가치: `BAL-MVP-001`
- 라운드별 등장과 요청 구성: `CNT-MVP-ROUND-001`
- 시설·일꾼·인벤토리의 작동 규칙: `SYS-CORE-001`
- 아이콘 조형·색·애니메이션: `UI-MVP-001` 및 MVP 아트 명세
- 판매 가격과 보상, 도구, 추가 성장 엔터티

## 3. 분류 규칙

MVP 아이템은 다음 세 분류만 사용한다.

| 분류 | 정의 | 납품 | 강화 |
|---|---|:---:|:---:|
| 원자재 `raw_material` | 공급함에서 직접 꺼내는 생산 입력 | 불가 | 불가 |
| 가공재 `processed_material` | 시설이 만든 중간 입력 | 불가 | 불가 |
| 장비 `equipment` | 의뢰가 요구할 수 있는 완성품 | 가능 | 가능 |

강화 단계는 별도 아이템 ID가 아니라 장비 인스턴스의 `enhancement_level` 속성이다. MVP에서 허용값은 `0`과 `1`이다.

## 4. 전체 아이템 목록

### 4.1 원자재

| ID | 표시명 | 역할 | 공급원 |
|---|---|---|---|
| `MAT_IRON_ORE` | 철광석 | 철 주괴의 원료 | `FAC_SUPPLY` |
| `MAT_WOOD` | 목재 | 목탄의 원료 | `FAC_SUPPLY` |
| `MAT_MANA_SHARD` | 마력 파편 | 강화석의 마력 성분 | `FAC_SUPPLY` |

원자재 인스턴스의 생성과 공급원 작동은 `SYS-CORE-001` §4~§5를 따른다.

### 4.2 가공재

| ID | 표시명 | 역할 | 소비처 |
|---|---|---|---|
| `MAT_IRON_INGOT` | 철 주괴 | 두 장비의 공통 제작 재료 | `FAC_WEAPON_BENCH` |
| `MAT_CHARCOAL` | 목탄 | 강화석 합성 재료 | `FAC_SYNTH_BENCH` |
| `MAT_ENHANCEMENT_STONE` | 강화석 | +1 강화 1회의 필수 소모품 | `FAC_ENHANCE_ANVIL` |

강화석은 선제 비축 판단을 만드는 핵심 중간재이며 자체로는 납품할 수 없다.

### 4.3 장비

| ID | 표시명 | 분류 | 강화 가능 | 요청 가능 단계 |
|---|---|---|:---:|---|
| `EQ_DAGGER` | 단검 | 무기 | 예 | 0, 1 |
| `EQ_IRON_SWORD` | 철검 | 무기 | 예 | 0, 1 |

두 장비는 제작 직후 항상 강화 단계 0이다. +1 결과도 같은 ID를 유지하고 `enhancement_level`만 1로 바뀐다.

## 5. 기자재와 일꾼 엔터티

기자재와 일꾼은 아이템처럼 이동하거나 인벤토리에 들어가지 않는다. 이 절은 ID·표시명·정체성만 소유하며 실제 상태 전이는 `SYS-CORE-001`을 따른다.

### 5.1 시설

| ID | 표시명 | 코어 역할 |
|---|---|---|
| `FAC_SUPPLY` | 재료 공급함 | 원자재 공급원 |
| `FAC_FURNACE` | 용광로 | 무인 변환 시설 |
| `FAC_WEAPON_BENCH` | 무기 제작대 | 작업자 처리 시설 |
| `FAC_SYNTH_BENCH` | 합성 작업대 | 작업자 처리 시설 |
| `FAC_ENHANCE_ANVIL` | 강화 모루 | 작업자 처리 시설 |
| `FAC_DELIVERY` | 납품대 | 납품 목적지 |
| `FAC_TRASH` | 쓰레기통 | 파기 목적지 |

영구 능력 `enhancement_capability_owned`가 가리키는 시설 집합은 합성 작업대와 강화 모루다. 라운드별 실제 사용 시설은 `CNT-MVP-ROUND-001`이 소유한다.

### 5.2 일꾼

| 카탈로그 ID | 표시명 | 역할 | 개별 인스턴스 ID |
|---|---|---|---|
| `ENT_WORKER` | 일꾼 | 작업자 처리 시설의 작업 한 건을 완료 시간까지 점유 | `WORKER_01`부터 순번 부여 |

MVP에는 능력·속도·직업이 다른 일꾼 유형이 없다. 실제 인스턴스 수는 `BAL-MVP-001`이 소유한다.

## 6. 아이템 상태 속성

- 모든 아이템은 §4의 `item_id`와 §3의 `category`를 가진다.
- 장비는 `enhancement_level` 0 또는 1을 가지며, 비장비는 항상 0이다.
- 아이템에는 품질, 내구도, 희귀도, 무게, 개별 가격, 무작위 옵션이 없다.
- 인스턴스의 위치·수명·소실 상태 전이는 `SYS-CORE-001`이, 실행 상태 타입은 `TECH-SIM-001`, 실제 JSON 직렬화 필드는 `TECH-DATA-001`이 소유한다.

## 7. ID 안정성 규칙

- 표시명이 바뀌어도 ID는 바꾸지 않는다.
- 하나의 ID가 두 정체성을 겸할 수 없고 아이템·시설·일꾼 ID는 서로 충돌할 수 없다.
- 강화 단계마다 별도 ID를 만들지 않는다.
- 레시피, 의뢰, 밸런스와 저장 데이터는 표시명이 아니라 ID를 참조한다.

## 8. 시스템 인터페이스

| 대상 | 이 문서가 제공하는 것 | 이 문서가 받는 것 |
|---|---|---|
| `SYS-CORE-001` | 유효 아이템·시설·일꾼 ID, 분류, 납품·강화 가능 여부 | 아이템 상태, 기자재 역할과 수명 규칙 |
| `CNT-RECIPE-001` | 입력·출력으로 사용할 ID | 생산 관계 |
| `CNT-MVP-ROUND-001` | 의뢰가 요청할 장비 ID | 사용 라운드와 요구 단계 |
| `BAL-MVP-001` | 값을 붙일 안정적인 ID | 작업·점수 파라미터 |
| `UI-MVP-001` | 표시명 키와 분류 | 아이콘·상태 표현 |
| `TECH-SIM-001` | 상태에 필요한 속성 목록 | 아이템 인스턴스와 위치 타입 |
| `TECH-DATA-001` | 카탈로그에 투영할 정의 | 실제 JSON 스키마와 참조 검증 |

## 9. 미결정 사항

| 질문 | MVP 처리 | 결정 방법 |
|---|---|---|
| 마력 파편과 강화석이 작은 화면에서 혼동되는가? | 별도 실루엣을 요구한다. | UI·아트 가독성 테스트 |
| +1 상태를 아이콘만으로 구분할 수 있는가? | 텍스트 없이도 구분 가능한 보조 표식을 요구한다. | `UI-MVP-001` 검증 |

## 10. 검증 방법

- 레시피와 의뢰의 모든 `item_id`가 §4에 존재해야 한다.
- 원자재 3종, 가공재 3종, 장비 2종 외 ID가 MVP 데이터에 있으면 실패다.
- 비장비 아이템의 강화 단계가 0이 아니면 실패다.
- 장비 강화 단계가 1을 넘으면 실패다.
- 납품 요청이 비장비 ID를 참조하면 실패다.
- 레시피와 라운드가 참조하는 모든 시설 ID가 §5.1에 존재해야 한다.
- 런타임 일꾼 인스턴스는 모두 `ENT_WORKER` 유형이어야 한다.

## 11. 완료 기준

- MVP에서 사용하는 아이템·시설·일꾼 전수가 중복 없이 열거되어 있다.
- 각 아이템의 분류·용도·납품·강화 가능 여부가 명확하다.
- 레시피·점수·라운드 배치가 문서에 섞여 있지 않다.
- `CNT-RECIPE-001`, `CNT-MVP-ROUND-001`, `BAL-MVP-001`이 표시명 추측 없이 ID를 참조할 수 있다.
