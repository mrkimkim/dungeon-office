## 1. 두 단계의 승인

이 문서는 사전 검증과 공개 출시를 구분한다.

| 단계 | 목적 | 승인에 필요한 것 |
|---|---|---|
| 사전 검증 후보 | 내부·폐쇄 테스터에게 5라운드의 기능과 재미를 검증한다. | 자동화, 실기, 5라운드 기능, 플레이테스트 게이트 |
| 공개 MVP 출시 | Google Play에서 Android용 무료 5라운드 앱을 공개한다. | 사전 검증 승인 + 서명·등록 정보·무료 가격 설정 승인 |

두 단계 모두 설치된 앱에서 R1~R5 전체가 무료로 열리며 앱 내부 결제, 페이월, 구매 복원, 상업 entitlement를 사용하지 않는다. 후속 유료 스테이지 가능성은 현재 후보의 기능 또는 테스트 경로가 아니다.

## 2. 검증 질문

MVP는 기능 수보다 아래 다섯 질문에 답해야 한다.

1. 드래그 운반을 기본으로 이해하면서도 필요할 때 두 탭 대체 조작으로 운반→착수→회수→납품의 인과를 유지할 수 있는가?
2. 일꾼과 시설을 병렬로 굴리는 선택이 손 빠르기보다 계획의 재미를 만드는가?
3. 과열, 시설 막힘, 의뢰 철회를 서로 다른 손실·병목·기회비용으로 읽는가?
4. 처리할 의뢰와 흘려보낼 의뢰를 고르는 트리아지가 억울하지 않은가?
5. 브리핑 예고를 읽고 미리 비축한 결과가 +1 강화 납품과 원정 결과의 성취로 이어지는가?

이 질문에 답하지 못하면 콘텐츠 양산이나 상용 기능보다 코어 규칙·밸런스 수정을 먼저 한다.

## 3. 5라운드 기능 인수 기준

| 라운드 | 검증 대상 | 통과 기준 |
|---|---|---|
| R1 | 기본 제작 사이클 | 내장 가이드만으로 공급→변환→제작→납품을 완료한다. 첫 성공에 필요한 정보와 피드백이 화면 안에 있다. |
| R2 | 병렬 작업·수납 | 두 일의 진행과 일꾼 점유를 구분할 수 있다. 산출물 방치는 시설만 막고, 수납하면 다음 작업을 이어갈 수 있다. |
| R3 | 과열·트리아지 | 과열 위험을 사전에 읽고 회수하거나 감수할 수 있다. 의뢰 철회는 감점·보유품 소실 없이 기회만 없애며 우선순위 선택이 실제 점수 차이를 만든다. |
| R4 | 예고→비축 | 브리핑의 예고와 실제 의뢰가 일치한다. 플레이어가 필요한 중간재나 장비를 미리 준비해 종반 의뢰에 사용할 수 있다. |
| R5 | 결정론적 +1 강화와 종합 마감 | 강화 자원 준비→장비 제작→+1 강화→납품이 한 판에서 연결된다. 강화는 같은 입력에서 항상 성공하고, 마감 전 완주가 결과 화면의 원정 성패로 이어진다. |

모든 라운드에서 일시정지 중 시간·작업·인내·과열이 진행되지 않아야 하며, 재개 후 정확히 다음 틱부터 이어져야 한다.

### 5.1 수동 실기 기록 양식과 위치

후보마다 연결된 GitHub Issue 하나를 `QA — <versionName> (<short commit SHA>)` 형식으로 만들고, 이 절의 체크리스트와 아래 표를 복사해 **비식별 결과 요약의 단일 기록 위치**로 사용한다. 후보 PR에는 해당 Issue를 연결한다. 자동화·에뮬레이터 결과로 물리 기기 항목을 체크하지 않는다.

| 기록 항목 | 값 |
|---|---|
| 후보 commit SHA / APK SHA-256 | |
| Godot / data / sim / save 버전 | |
| 기기 모델 / Android 버전 / API / 화면 형태 | |
| 설치 방식 / 오프라인 상태 | |
| 시작·종료 시각 / 콜드 스타트 / 피크 메모리 / 설치 크기 | |

실행 시작 전 `./tools/check_env.sh --require-device`와 debug APK 검증을 통과시키고, 기기에서는 신규 설치와 업데이트 설치를 각각 확인한다. 기기 serial, 계정 정보, 사용자 저장 원문과 서명 비밀은 Issue·PR·저장소에 기록하지 않는다.

## 6. 성능·표현 최소 기준

- 최소 사양 기기에서 플레이 중앙 프레임률 60fps를 목표로 하며, 긴 프레임의 95백분위가 33.3ms를 넘지 않는다.
- 프레임 드랍이나 앱 인터럽트가 시뮬 틱 결과를 바꾸지 않는다.
- 피크 메모리 400MB 이하, 설치 크기 200MB 이하, 탭→타이틀 콜드 스타트 3초 이하를 실기에서 기록한다.
- 플레이 화면의 필수 조작은 세로 한 화면 안에 있고 스크롤·핀치를 요구하지 않는다. 아이템 운반은 드래그를 기본으로 하되 모든 운반에 두 탭 대체 조작이 있어 드래그만을 유일한 입력 경로로 강제하지 않는다. 전체 화면 제작 체인의 세로 스크롤은 이 플레이 무스크롤 기준의 예외다.
- 모든 드래그 출발지·드롭 목적지·탭 대상은 44pt 이상이며 안전 영역 밖에 필수 조작을 두지 않는다.
- `ART-MVP-001`의 시설 7종과 아이템 8종은 최소 표시 크기와 흑백 캡처에서 이름 없이도 역할·실루엣을 구분할 수 있다.
- 정상 화염의 오렌지와 과열 임박의 레드+점멸, 작업 스카이, 완료 골드, 강화 퍼플을 서로 혼동하지 않는다.
- 현재 1차 패스의 개별 PNG가 최소 사양의 프레임·메모리·설치 크기 기준을 넘기면 아틀라스 패킹보다 먼저 해상도·import·표시 수를 측정하고 조정한다.
- 과열·완료·착수 거절은 색 하나나 소리 하나에만 의존하지 않고 형태·텍스트·소리 또는 햅틱 중 둘 이상으로 전달한다.
- 음소거와 햅틱 끔 상태에서도 모든 판정 정보가 화면에 남는다.
- 앱에 원격 분석·광고·앱 내부 결제 SDK가 없어야 하며, 의도하지 않은 네트워크 요청이 없어야 한다.
- 앱 내부 법적 고지는 네트워크를 사용하지 않고 큰 글씨와 긴 URL에서도 가로 잘림 없이 읽을 수 있어야 한다.

## 7. 핵심 재미 플레이테스트

### 7.1 대상과 방식

- 개발에 참여하지 않은 첫 플레이어 최소 5명
- 기존 게임 규칙을 사전 설명하지 않고 빌드 내 브리핑·온보딩만 제공
- 동일 후보 빌드와 신규 프로필 사용
- 진행 관찰표, 사후 인터뷰, 동의한 화면 녹화만 사용
- 퍼널·행동 분석 SDK는 사용하지 않는다.

진행자는 장애·크래시 외에는 개입하지 않는다. 개입이 있었다면 해당 참가자의 “무개입 완주” 결과와 분리해 기록한다.

### 7.2 통과 기준

- 5명 모두 외부 설명 없이 R1의 첫 납품을 완료한다.
- 최소 4명이 첫 안내 뒤 아이템 드래그로 공급→시설→인벤토리 또는 납품 이동을 수행하고, 무효 드롭 뒤 아이템이 보존됐음을 이해한다.
- 최소 4명이 드래그가 어려울 때 두 탭 대체 조작을 발견하거나 안내 뒤 사용할 수 있고, 두 입력을 서로 다른 게임 규칙으로 오해하지 않는다.
- 최소 4명이 활성 의뢰의 `제작법`을 열어 목표 장비까지 필요한 원자재·중간재·시설을 설명하고, 닫은 뒤 시간이 열람 중 흐르지 않았음을 확인한다.
- 최소 4명이 45분 안에 R5까지 도달하고, 실패 시 스스로 재도전해 완료한다.
- 최소 4명이 R3에서 일부 의뢰를 의도적으로 흘려보내는 우선순위 선택을 말이나 행동으로 보인다.
- 최소 4명이 R4~R5의 브리핑 예고를 근거로 필요한 재료·장비를 미리 만들거나, 첫 실패 뒤 같은 계획을 스스로 수정한다.
- 최소 4명이 R5의 +1 강화를 확률 시도가 아닌 준비 공정으로 이해하고 완주한다.
- 최소 4명이 “결과가 운보다 내 작업 순서에서 나왔다”고 답한다.
- 최소 4명이 자신의 납품 중 하나와 결과 화면의 원정 성패를 연결해 설명한다.
- 최소 3명이 테스트 종료 전 한 라운드를 자발적으로 다시 플레이하거나, 즉시 다시 해 보고 싶은 구체적 개선 계획을 말한다.
- 과열·철회·시설 막힘을 같은 종류의 임의 처벌로 오해하는 참가자가 1명을 넘지 않는다.

수치 통과 여부와 별개로 반복되는 혼란·지루함·신체적 불편은 원문과 발생 라운드를 함께 기록한다.

### 8.2 공개 MVP 출시 승인

공개 MVP 출시는 다음을 모두 만족해야 한다.

1. §8.1의 사전 검증 후보가 승인되었다.
2. Google Play 앱 가격이 무료로 설정되고 `BUS-MVP-001`의 R1~R5 무료 정책과 일치한다.
3. 서명된 최종 Android AAB가 §5 행렬을 통과한다.
4. Google Play 설명·Android 앱 아이콘·실제 후보 빌드 스크린샷·연령·개인정보 표시가 5라운드 앱과 일치하고 필수 규격·수량을 충족한다.
5. Google Play 심사를 통과해 공개 가능한 상태다.
6. Play Console에 현재 앱용 유료 상품이 없고, 최종 빌드에 Billing SDK·상품 ID·페이월·구매 복원·상업 entitlement 경로가 없음을 확인한다.
7. `REL-PLAY-001`의 공개 지원·개인정보·오픈소스 URL이 인증 없이 HTTPS로 열리고 앱 내부 원문과 일치한다.
8. Play Console의 Data safety가 `수집 없음·공유 없음·계정 없음·광고 없음·앱 내부 결제 없음`이라는 최종 AAB의 실제 동작과 일치한다.
9. Google Play 아이콘 512×512, 기능 그래픽 1024×500과 실제 후보 휴대전화 스크린샷 최소 2장이 `ART-MVP-001` 규격을 통과한다.
10. application ID, versionCode, target API, upload 인증서 지문과 Play App Signing 상태를 릴리스 기록에 남기며 키·암호 자체는 기록하지 않는다.
11. 현재 개발자 계정에 Play Console이 요구하는 신원·실기기 인증, 비공개 테스트와 프로덕션 액세스 과제를 완료했다.

현재 MVP는 무료 앱이므로 앱 내부 결제 테스트를 수행하지 않는다. 후속 유료 스테이지가 승인되면 기존 게이트를 억지로 확장하지 않고 해당 버전의 결제·복원·오프라인 권한 QA를 별도로 명세한다.

## 10. 완료 조건

- R1~R5 기능·결정론·저장·실기와 핵심 재미 게이트가 모두 실행 가능한 테스트와 결과 양식을 가진다.
- `QA-REL-01~11`이 최종 서명 AAB와 동일 Git 커밋·Play Console 계정을 대상으로 통과한다.
- 공개 URL, 앱 내부 고지, Play Console 제출과 최종 AAB의 제품·데이터 경계가 서로 일치한다.
- 드래그 우선·두 탭 대체 운반, 무효·화면 밖 드롭 취소와 출발지 보존, 실제 인벤토리·`FAC_DELIVERY`·`FAC_TRASH` 목적지와 제작법 일시정지·재개 체인이 자동화와 실기에서 모두 검증된다.
- P0·P1이 0건이고, 남은 P2·P3는 책임자·수정 목표 버전·사용자 영향을 기록한다.
- 비밀 키, 키 암호, 계정 복구 정보는 QA 결과나 저장소에 포함하지 않는다.
