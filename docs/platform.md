## 1. 엔진·언어

### 1.1 결정: Godot 4.4 안정 계열 + 타입드 GDScript

| 후보 | 판단 |
|------|------|
| **Godot 4.4.x** | **채택.** MIT 라이선스 — 매출·설치 수와 무관하게 비용 0, 약관 변경 리스크 0 (무료 배포+본편 언락 IAP라 설치 수가 매출과 어긋나며, 러닝 로열티·매출 연동 구조는 원가를 왜곡한다). 2D 전용 파이프라인이 가볍고, `--headless` 실행이 1급 기능이라 결정론 테스트(§3)·CI(§7)와 궁합이 좋다. 4.x는 2026년 기준 모바일 익스포트(Android AAB, iOS Xcode 프로젝트)가 충분히 성숙했다. |
| Unity 6 | 기각 — 기능은 충분하나 2023 Runtime Fee 사태가 보여준 약관 변동 리스크, 무거운 에디터·빌드 체인이 1인 개발의 반복 속도를 깎는다. 이 게임은 Unity의 강점(3D·에셋스토어 미들웨어)을 쓰지 않는다. |
| Flutter + Flame | 기각 — UI는 강하나 게임 필(파티클·애니메이션·사운드 믹싱) 툴링이 약해 "망치질 손맛"(GDD §5.1)에 공수가 더 든다. |
| 네이티브 (SpriteKit + Kotlin) | 기각 — 결정론 코어를 두 번 구현·검증해야 한다. 1인 개발에서 즉시 탈락. |

- **버전 고정**: Godot 4.4 안정 계열의 특정 패치 버전을 `project.godot`·CI·익스포트 템플릿까지 일치시켜 고정한다. 마이너 업그레이드(4.5 등)는 마일스톤 경계에서만, 골든 리플레이 테스트(§7.4) 통과를 조건으로 수행한다.
- **언어: 타입드 GDScript 단일**. C#(.NET)은 기각 — iOS .NET 익스포트가 아직 2급 지원이고, 언어 두 개는 1인 팀의 사치다. 성능 우려는 없다: 시뮬레이션 규모가 일꾼 ≤6, 시설 ≤10종, 의뢰 슬롯 수 개로 상수 수준이다. 시뮬 코어의 수치는 전부 정수라(§3.2) GDScript로도 결정론이 성립한다.
- **렌더러: Compatibility (OpenGL ES 3.0)**. 2D 세로 게임에 Vulkan(Mobile 렌더러)의 이점이 없고, 구형 기기 호환과 배터리에서 유리하다.

## 2. 최소 지원 사양

### 2.1 OS·기기
- **Android**: 최소 **Android 8.0 (API 26)**, OpenGL ES 3.0, RAM 2GB. 타깃 SDK는 Play 스토어 요구 최신을 따른다(심사 요건, 매년 갱신).
- **iOS**: 최소 **iOS 15.0** (2026년 기준 활성 기기 대부분 커버, 구형 대응 비용 절감).
- 판단 근거: 무료 체험 모델이라 다운로드층에는 저가·구형 기기가 포함되지만, **본편 언락 구매 결정은 체험이 쾌적했던 기기에서 나온다** — 지원 하한은 "체험이 쾌적한 범위"의 바닥에 맞춘다. 그 아래 기기까지 QA 매트릭스를 넓히는 것은 구매로 이어지지 않는 층에 비용을 쓰는 일이다.

### 2.2 화면 대응 (세로 고정)
- **기준 캔버스 1080×1920 (9:16), 세로 고정**. 스케일 모드 `canvas_items` + `expand`: 세로로 긴 기기(9:19.5~9:22)는 위아래로 배경·여백이 늘어나고, HUD는 세이프 에어리어 앵커에 붙는다.
- **노치·펀치홀**: 모든 인터랙티브 UI는 OS 세이프 에어리어 내부에 배치(엔진 API 사용). 배경 아트만 풀블리드.
- **태블릿**: 플레이 영역을 최대 폭(기준 폭의 1.0배)으로 고정하고 좌우는 대장간 테마 필러박스. 근거 — GDD 원칙 1 "한 손, 한 화면"의 터치 거리 보존이 화면 활용보다 우선.
- **폴더블**: 특별 취급하지 않는다. 화면 크기 변경(접힘/펼침) 이벤트 시 자동 일시정지(§3.3의 앱 이탈 정지와 동일 경로) 후 재레이아웃. 접힘 상태는 폰, 펼침 상태는 태블릿 규칙을 따른다.

### 2.3 성능 예산
- **프레임**: 최소 사양 기기에서 60fps 목표, 옵션으로 30fps 배터리 모드 제공. 시뮬레이션은 렌더 프레임과 분리된 고정 틱(§3.1)이므로 프레임 드랍이 판정에 영향을 주지 않는다.
- **배터리**: 라운드 밖 화면(지도·상점·장부)은 저전력 모드(low processor mode) + 프레임 상한 30. 일시정지 화면도 동일.
- **앱 용량**: 설치 기준 **150MB 이하 목표, 200MB 상한**. 스토어 다운로드 100MB 이하 지향. 2D 스프라이트 아틀라스 + OGG 오디오면 여유 있다.
- **메모리**: 피크 400MB 이하 (2GB 기기에서 OS 킬 회피).
- **콜드 스타트**: 탭 → 타이틀 3초 이내 (세션 완결성 원칙 9 — "잠깐 한 판"을 막는 로딩은 적이다).

## 3. 결정론적 시뮬레이션 구현 노트

GDD 원칙 4(확률 판정 없음 — 무작위는 의뢰 등장 변주에만)와 §3.3(언제든 일시정지, 중단 저장 후 정확히 그 순간부터 재개)을 코드 구조로 보장한다.

### 3.1 고정 틱 — 결정
- **시뮬레이션은 20Hz 고정 틱(1틱 = 50ms)**. 델타타임 누적은 기각 — 부동소수 누적 오차와 기기별 프레임 편차가 "같은 판단 = 같은 결과"를 깨뜨린다.
- 모든 게임 시간(납품기한, 작업 시간, 과열 유예, 인내 게이지)은 **틱 정수**로 정의한다. 라운드 2~4분 = 2,400~4,800틱.
- 렌더 루프는 자유 fps로 돌며 틱 사이 게이지를 **보간 표시만** 한다. 보간은 표현이지 상태가 아니다.
- 탭 입력은 도착한 다음 틱에 커맨드로 처리(50ms 양자화 — 체감 불가, GDD §5.1 "타이밍 판정 없음"이라 손해도 없다).
- **일시정지 = 틱 공급 중단**. 상태 변화가 원천적으로 없으므로 "재개하면 그 순간부터"가 자동 성립. 정지 중 관찰·계획 허용(§3.3)은 뷰가 시뮬 상태를 계속 읽기만 하면 된다.

### 3.2 상태 규율
- 시뮬 상태에 **부동소수점 금지**. 시간=틱, 점수=정수, 게이지=천분율 정수. 언어·CPU의 float 재현성 문제를 원천 차단한다.
- 시뮬은 순수 함수 `step(state, commands[]) -> state'`로 구현. 엔진 노드·전역·시계·`randi()` 참조 금지 (§6.2에서 구조로 강제).
- **직렬화 범위 = 시뮬 상태 전부**: 현재 틱, RNG 내부 상태, 의뢰 생성기 커서, 활성 의뢰(점수·남은 인내), 시설별 상태(진행 중 레시피·남은 틱·과열 카운트다운), 일꾼 배정, 인벤토리 내용, 납품 누계 점수, 커맨드 일련번호. 예상 크기 수 KB — 어느 시점이든 통째로 스냅샷 가능하다.

### 3.3 무작위성 — 시드 정책 (결정)
- RNG는 자체 구현 **PCG32** 하나. 유일한 소비자는 **의뢰 생성기**(라운드 데이터가 정의한 풀에서의 추첨·간격 변주)다. 그 외 어디서도 RNG를 호출하지 않는다 — 승패를 가르는 주사위 금지(원칙 4)를 호출 지점 단위로 보장.
- **시드 = hash64(round_id, attempt_index)**. `attempt_index`는 라운드별 도전 횟수 카운터(영구 저장).
  - **재도전 변주 정책**: 재도전마다 시드가 바뀐다 — 변주는 콘텐츠이지 암기 퍼즐이 아니며, "의뢰 순서 외우기"가 아니라 "판을 읽는 계획"(GDD §1.3)이 실력이 되게 한다. 단 브리핑 예고 의뢰·스크립트 고정 의뢰(라운드 데이터의 `script` 항목)는 시드와 무관하게 항상 등장하므로, 재도전 계획의 뼈대는 유지된다.
  - 같은 (round_id, attempt_index)는 어느 기기에서든 같은 판을 재현한다 — 버그 리포트·리플레이·밸런스 봇에 필수.
  - 튜토리얼성 라운드는 데이터에서 `pools`를 비우고 전량 스크립트로 짜서 변주 0으로 만들 수 있다 (설계서 재량).
- **중단 저장**은 RNG 내부 상태까지 통째로 저장하므로, 재개 후의 의뢰 변주도 중단 전과 완전히 동일하다.

### 3.4 리플레이 (파생 이득)
- `시드 + 틱 스탬프 커맨드 로그`만으로 라운드 전체가 재현된다. 리플레이 파일은 QA 재현, 골든 테스트(§7.4), 밸런스 봇(§7.5)의 공용 포맷이다. 별도 공수 최소 — 결정론의 공짜 배당이다.

## 4. 데이터 주도 콘텐츠 파이프라인

GDD 원칙 6: "배울 것은 늘지 않고 만들 것만 늘어난다." 콘텐츠는 전부 데이터 파일이고, 코드는 4계열 시설의 규칙 엔진만 안다.

### 4.1 포맷 — 결정: JSON
- 런타임 포맷은 **JSON** (UTF-8, `res://data/`에 평문 동봉). 근거: Godot 내장 파서, 사람이 읽고 diff 가능, 스키마 검증 용이. CSV 기각 — 중첩 구조(레시피 입력 배열, 라운드 스크립트) 표현 불가. 커스텀 포맷 기각 — 1인 개발에 파서 유지비는 낭비.
- **밸런스 수치서와의 연결**: 모든 튜닝 수치는 `data/balance.json`과 각 콘텐츠 파일에만 존재한다. **시뮬 코드에 게임플레이 상수 하드코딩 금지**(리뷰 체크리스트 항목). 밸런스 문서의 수치 개정 = JSON 교체 = 코드 수정 0. 개발 빌드는 데이터 핫리로드를 지원해 튜닝 루프를 초 단위로 만든다.
- 모든 표시 문자열은 데이터에 키만 두고 Godot 번역 CSV로 분리 (한/영 현지화 대비).

### 4.3 컴파일·검증 단계
`tools/validate_data.gd`(헤드리스 실행)가 커밋마다 CI에서, 그리고 디버그 빌드 부팅 시 실행된다.
- **참조 무결성**: 모든 id 상호 참조, 고아 데이터 검출.
- **GDD 제약의 기계 검증**: 계열별 시설 종수 상한(3/3/3/1), 장비 분류 3종, 생산 규칙(§8.1) 2종 외 금지, 전략 자원 1종, 강화석-강화 시설 해금 동기화(§8.4-6), 다인 작업 의뢰는 일꾼 3명 전제 구간 이후(§6), **누적 별 게이트 산정**(각 스테이지 `unlock.cumulative_stars_required` ≤ 그 시점까지 도달 가능한 별 총량 — "3성 강제 없이 도달 가능한 별 이하", balance §6).
- **입장 요건 자동 도출**: 라운드의 script+pools에서 도달 가능한 전체 의뢰(단골 특별 의뢰 `is_regular` 포함)를 수집하고, 각 의뢰 장비의 공정 사슬을 **재귀로** 걷는다 — 조립품은 `in`의 부품을, 부품은 다시 그 `in`의 재료·레시피를 따라가 필요 설비 집합을 계산해 라운드 데이터에 파생 필드로 주입한다. §3.1 "라운드 안의 모든 의뢰는 정의상 이행 가능"이 사람 실수가 아니라 파이프라인으로 보장된다.
- 41 라운드 규모에서는 스프레드시트 중간 단계 없이 JSON 직접 편집 + 검증기로 충분하다. 물량이 늘면 그때 시트→JSON 변환 스크립트를 추가한다 (지금 만들지 않는다).

## 5. 세이브 시스템

### 5.1 두 개의 저장물
| 파일 | 내용 | 수명 |
|------|------|------|
| `profile.json` | 영구 상태(§7.3): 골드, 일꾼 수, 시설(종류·수·등급), 도구, 설치 슬롯 확장 단계, 인벤토리 확장 단계, 라운드별 별 최고 기록, 계약·장부, 라운드별 attempt_index, **본편 언락 상태(entitlement) + IAP 영수증**, 설정, 통계 | 영구 |
| `round_snapshot.json` | 중단된 라운드의 시뮬 상태 전체(§3.2) + 시드 + 커맨드 로그 | 라운드 종료 시 삭제 |

- 라운드마다 리셋되는 것(재료·장비·강화석·인벤토리·의뢰·점수)은 profile에 절대 넣지 않는다 — GDD §7.3의 이원 구조를 파일 경계로 물화한다.

### 5.2 저장 시점·안전성
- **profile 저장**: 라운드 정산 직후, 상점 구매 직후, 앱 백그라운드 진입 시.
- **snapshot 저장**: 라운드 개시 시, 앱 백그라운드/일시정지 진입 시, 라운드 중 30초 주기 자동 (크래시 대비 — 상태가 수 KB라 비용 무시 가능). §3.3 "앱 이탈 시 자동 정지·저장"의 구현이다.
- **원자적 쓰기**: 임시 파일 작성 → fsync → rename. A/B 이중 슬롯 + SHA-256 체크섬으로 손상 시 직전 슬롯 복구. 결제 1회(본편 언락 IAP) 게임에서 세이브 파손은 환불 사유이자 결제한 본편의 유실이다 — 여기는 과잉 설계가 정당하다.
- **언락 상태는 세이브 최우선 보호 대상이다.** entitlement·IAP 영수증(§5.1)의 유실은 결제한 본편의 재잠금을 뜻하므로 A/B 이중화·체크섬 보호를 최우선으로 적용한다. 그럼에도 로컬 유실이 나면 최종 복구 경로는 §9의 **구매 복원**이다.
- **오프라인 언락 보장**: 구매·복원 성공 시 영수증을 로컬에 저장하고, 이후 본편 개방은 **로컬 상태(entitlement)만 참조**한다 — 매 실행 시 네트워크 확인이 불필요해 기내 모드에서도 본편이 열린다 (monetization §2.3).
- 세이브의 암호화·난독화는 하지 않는다. 무료 배포+본편 언락 IAP(결제 1회)의 오프라인 싱글 게임에서 세이브 조작은 본인 소관이다 (§7.3의 오프라인 원칙과 같은 철학).

### 5.3 버전 마이그레이션
- 두 파일 모두 최상위에 정수 `save_version`. 로드 시 `migrate_N_to_N+1()` 체인을 순차 적용. 마이그레이션 함수는 삭제하지 않는다 (출시 후 어떤 구버전에서도 최신으로).
- 각 릴리스의 실제 세이브 파일을 `tests/fixtures/saves/`에 보관하고, CI가 전 버전 → 최신 마이그레이션을 회귀 테스트한다.
- 스냅샷은 마이그레이션하지 않는다 — 앱 업데이트로 시뮬 로직이 바뀌면 진행 중 라운드의 결정론을 보장할 수 없으므로, 버전 불일치 스냅샷은 폐기하고 해당 라운드를 브리핑부터 다시 시작하게 한다 (라운드는 2~4분 — 수용 가능한 손실, 별 기록은 profile에 있어 무손실).

### 5.4 클라우드 세이브 — 결정: v1.0 미탑재, 1.x 옵트인 도입
- v1.0은 로컬 저장만. 근거 — 서버 없음·완전 오프라인 원칙 하에서 플랫폼 동기화(iCloud/Play Games)는 별도 플러그인·계정 UI·충돌 해소 UX를 요구하며, 1인 개발의 출시 리스크로 부적절.
- 단, 세이브를 단일 디렉터리·소용량·자기완결로 설계해 두었으므로 1.x에서 iCloud Documents(iOS)·Play Games Saved Games(Android)를 **옵트인**으로 얹는 비용이 낮다. 도입 시에도 로컬이 항상 진실이고 클라우드는 백업이다 — 오프라인 플레이는 어떤 경우에도 막히지 않는다.

## 6. 프로젝트 구조

### 6.1 디렉터리
```
res://
  sim/          # 순수 시뮬레이션 — 엔진 비의존 (RefCounted만, Node 금지)
    state.gd  step.gd  commands.gd  rng.gd  request_gen.gd  scoring.gd
  game/         # 표현·입출력 — sim을 소유하고 구동
    scenes/     # boot / title / map(가도) / forge(상점·배치) / round / result / ledger(장부)
    view/       # 시뮬 이벤트 → 애니메이션·사운드 바인딩
  data/         # §4의 JSON + 번역 CSV
  tools/        # validate_data.gd, max_score.gd, bot_player.gd (전부 헤드리스 실행 가능)
  tests/        # 유닛 테스트, 골든 리플레이, 세이브 픽스처
```

### 6.2 시뮬·뷰 분리 규율
- `sim/`은 **Node·Input·씬 트리·엔진 RNG·OS 시계 참조 금지**. 입력은 커맨드 객체, 출력은 새 상태 + 틱별 이벤트 목록(작업 완료, 과열 경고, 의뢰 도착/철회, 납품 매칭). CI에 정적 검사(금지 API grep)를 걸어 규율을 기계로 지킨다.
- `game/round` 씬은 매 렌더 프레임에 경과 실시간을 틱으로 환산해 `step()`을 0회 이상 호출하고, 이벤트를 연출로 번역한다. 일시정지는 이 펌프를 멈출 뿐이다.
- 이 분리 덕에 `godot --headless`로 시뮬만 실행하는 결정론 테스트·밸런스 봇이 렌더링 없이 초당 수천 라운드를 돌 수 있다.
- 씬 흐름: `boot → title → map ⇄ (forge | ledger) → round(브리핑 오버레이 → 플레이 → result) → map`. 브리핑·결과는 round 씬 내부 상태로 두어 재도전 즉시 재시작(§11.1, 브리핑 생략)을 씬 리로드 없이 처리한다.
- **페이월(언락 게이트)의 씬 편입**: 체험 범위는 거래처 1이다. 미언락 상태로 거래처 2 이상에 입장하려 하면(노출 지점 = 거래처 1의 첫 계약 체결 직후, monetization §2.2) map 위에 **오버레이 페이월**을 띄운다 — 별도 씬이 아니라 map의 모달이다. 구매 또는 구매 복원(§9)이 성공하면 entitlement(§5.1)를 세우고 즉시 잠금을 해제해 진행을 잇는다. 결제 상호작용 자체는 OS 스토어 UI로 위임되므로 씬 그래프는 바뀌지 않는다.

## 7. 빌드·배포·품질

### 7.1 버전 관리
- Git, **트렁크 기반**: `main` 단일 영구 브랜치 + 수명 짧은 작업 브랜치. 1인 개발에 git-flow는 기각 — 의식(儀式)만 남는다. 릴리스는 태그(`v0.3.0`)로. 문서(GDD·본 문서·하위 설계서)도 같은 리포지토리에서 버전 관리한다.

### 7.2 CI 최소 구성 (GitHub Actions)
- **커밋/PR마다** (Linux 러너, 수 분): 데이터 검증(§4.3) → 시뮬 유닛 테스트 → 골든 리플레이(§7.4) → 밸런스 회귀(§7.5) → 세이브 마이그레이션 테스트(§5.3) → **언락 상태 회귀**(미언락 시 거래처 2+ 입장 차단, 구매·복원 후 개방, 로컬 저장 영수증만으로 네트워크 차단[기내 모드] 개방 — 결제 SDK는 목(mock)으로 대체).
- **태그 시**: Android AAB 익스포트(Linux 러너) + iOS Xcode 프로젝트 익스포트·아카이브(macOS 러너). 서명 키는 GitHub Secrets. 스토어 업로드는 초기엔 수동, 알파부터 fastlane 자동화.
- 익스포트 템플릿 버전을 엔진 버전과 함께 고정·캐시한다.

### 7.3 크래시 리포팅·분석 — 완전 오프라인 원칙과의 절충
- **앱에 네트워크 SDK를 넣지 않는다.** Firebase/Sentry류는 기각 — 상시 통신은 "완전 오프라인"을 마케팅 문구에서 거짓말로 만들고, 개인정보 고지 부담만 늘린다.
- **예외 — 결제·복원만은 스토어 OS API로 통신한다.** 본편 언락 IAP의 구매·구매 복원은 StoreKit(iOS)·Play Billing(Android)으로만 오가며, 이는 게임플레이·저장·텔레메트리의 완전 오프라인과 **별개 층**이다(§9). 게임 로직·세이브는 여전히 네트워크에 닿지 않고, 언락 이후 본편 개방도 로컬 상태만 참조한다(§5.2). 아래의 **텔레메트리·분석 SDK 금지 원칙은 그대로 유지된다** — 결제 API는 분석 채널이 아니다.
- 크래시 수집은 **OS 레벨 옵트인 채널**만 쓴다: Google Play Console(Android vitals)·App Store Connect 크래시 리포트 — 사용자가 OS 설정에서 동의한 경우에만 플랫폼이 수집하며, 앱은 코드 0줄.
- 보조 수단: 로컬 순환 디버그 로그(기기 밖 전송 없음) + 설정 화면의 "문의 메일에 로그 첨부" 버튼. 전송은 사용자의 명시적 행동 = 완전한 옵트인.
- 행동 분석(퍼널·리텐션 텔레메트리)은 **탑재하지 않는다**. 밸런스 판단은 §7.5의 시뮬레이션 도구와 베타 테스터 피드백으로 한다.

### 7.4 결정론 자동 테스트
- **골든 리플레이**: 대표 라운드들의 `시드+커맨드 로그` → 최종 상태 해시를 저장해 두고, 모든 커밋에서 재실행 해시 일치를 검증. 엔진 업그레이드·리팩터링이 결정론을 깨면 즉시 적발된다. x86(CI)과 ARM(실기 스모크)에서 동일 해시 — 정수 전용 상태(§3.2)라 성립한다.
- 스냅샷 왕복 테스트: 임의 틱에서 저장→로드→계속 실행한 결과가 무중단 실행과 해시 일치.

### 7.5 밸런스 회귀 — 라운드 이론 최대점 계산기
- `tools/max_score.gd`: 라운드 데이터로부터 **이론 최대점 상한**을 계산한다 — 납품기한 내 일꾼-시간 총량, 시설별 처리량, 의뢰 등장 시각·인내로 제약을 완화한 상한 산출(정확한 최적 스케줄이 아니라 보수적 상한 — 구현 하루짜리).
- `tools/bot_player.gd`: 탐욕 전략 봇이 헤드리스로 라운드를 실제 플레이해 **달성 가능 하한**을 산출. 변주 검증을 위해 시드 100개 일괄 실행.
- CI 검증식 (수치는 밸런스 문서가 소유, 검증식만 여기서 확정): `실패선 < ★1 ≤ 봇 하한`(권장 수준 설비로 클리어 가능), `★3 ≤ 이론 상한 × 여유율`(3성이 수학적으로 가능), 누적 별 게이트 ≤ "3성 강제 없는" 도달 가능 별 총량(A-3 제약). 밸런스 JSON을 고치는 모든 PR이 이 검증을 자동 통과해야 한다.

## 9. 미결(본 문서 v0.2에서 확정할 것)
- **결제·복원 연동은 필수 구현이다** (무료+본편 언락 IAP 모델). StoreKit(iOS)·Play Billing(Android)으로 **비소모성 언락 상품 1종 + 구매 복원 버튼**(양 스토어 심사 요건)을 구현하며, Godot 결제 플러그인의 소싱·검증은 **M1에 확정**(§8)한다. 남은 미결은 구체 플러그인 선정과, 이와 **별개 선택 사안**인 Game Center/Play Games 업적 도입 여부뿐이다.
- 태블릿 필러박스 아트의 외주 범위 (M1에서 화면 설계서와 함께).
- 1.x 클라우드 세이브(§5.4)의 충돌 해소 정책 상세.

*v0.1 — 테크 리드. 본 문서의 모든 결정은 GDD v1.9를 전제로 하며, GDD 개정 시 §3(결정론)·§4(데이터 제약 검증기)가 최우선 재검토 대상이다.*
