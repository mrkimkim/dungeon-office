## 1. 목적과 범위

이 문서는 승인된 MVP 규칙과 콘텐츠에 값을 부여하는 유일한 Source of Truth다. 다른 문서는 파라미터 이름과 의미만 정의하고 실제 숫자를 복제하지 않는다.

이 수치 세트는 최종 출시 밸런스가 아니라 다음 가설을 검증하기 위한 첫 플레이 가능 기준선이다.

- R1에서 한 번의 생산·납품 사이클을 빠르게 이해할 수 있는가?
- R2에서 무인 용광로와 일꾼 작업의 병렬성이 생기는가?
- R3에서 과열과 트리아지가 선택을 만들되 한 번의 실수로 실패를 강제하지 않는가?
- R4에서 예고를 읽고 선제 비축하는 것이 실제 이득인가?
- R5에서 강화석과 장비를 미리 준비해 +1 프로젝트를 완주하는가?

## 2. 명시적 비범위

- 규칙, 아이템, 레시피, 라운드 콘텐츠의 추가·변경
- 화면 애니메이션 시간과 오디오 타이밍
- 렌더 프레임, 고정 틱, 정수 직렬화 방식: `TECH-SIM-001`, `TECH-DATA-001`
- 외부 결제와 상업 모델
- MVP 이후의 수치나 성장 곡선

## 3. 단위와 반올림

- 게임 시간의 규범 단위는 초다.
- 아래 값은 모두 정확한 값이며 `TECH-SIM-001`의 20Hz 시간 단위로 손실 없이 변환해야 한다.
- 점수와 골드는 정수다.
- 배율 계산은 사용하지 않으므로 런타임 반올림 규칙이 없다.
- 플레이테스트 조정은 작업·인내·공개 시점은 1초 단위, 점수·골드는 5 단위로 한다.

| 파라미터 | 값 |
|---|---:|
| 일꾼 수 | 2명 |
| 시설별 설치 수 | 각 1대 |
| 인벤토리 용량 | 4칸 |
| 아이템 스택 크기 | 1 |

일꾼·시설·인벤토리 값은 R1~R5에서 변하지 않는다. R5에서 합성 작업대와 강화 모루만 영구 설비 구매로 추가된다.

### 4.1 상태 경고 임계값

| 파라미터 ID | 값 | 적용 경계 |
|---|---:|---|
| `overheat_danger_at` | 3초 | 과열 잔량이 3초 이하이고 0초보다 클 때 임박 |
| `request_urgent_at` | 10초 | 활성 의뢰 인내가 10초 이하이고 0초보다 클 때 임박 |
| `deadline_warning_at` | 30초 | 라운드 기한이 30초 이하이고 0초보다 클 때 경고 |
| `deadline_countdown_at` | 10초 | 라운드 기한이 10초 이하이고 0초보다 클 때 카운트다운 |

경계 진입 이벤트의 정확한 발행 순서와 한 번만 발행하는 규칙은 `TECH-SIM-001` §9가 소유한다. 값은 다음 관계를 만족한다.

```text
0 < overheat_danger_at < 용광로 과열 유예
0 < request_urgent_at < MVP 의뢰 인내의 최솟값
0 < deadline_countdown_at < deadline_warning_at < MVP 라운드 기한의 최솟값
```

## 6. 요청 점수와 인내

### 6.1 점수 구성

MVP 요청 점수는 다음 세 값의 합으로 정한다.

| 요소 | 값 |
|---|---:|
| 단검 기본점 | 10점 |
| 철검 기본점 | 20점 |
| +1 요구 추가점 | +30점 |

## 9. 경제 수치

### 9.1 라운드 완료 보상

| 결과 | 반복 완료 보상 |
|---|---:|
| 실패 | 0G |
| 별 1 | 20G |
| 별 2 | 30G |
| 별 3 | 45G |

최초 별 1 이상 클리어 보너스는 라운드당 `20G`다.

### 9.2 상점

| 상품 ID | 가격 | 노출 |
|---|---:|---|
| `SHOP_ENHANCEMENT_KIT` | 150G | R4 최초 클리어 뒤 |

R1~R4를 각각 별 1로 최초 클리어했을 때의 최소 누적 골드는 다음과 같다.

```text
4 × (별 1 보상 20G + 최초 클리어 20G) = 160G
160G >= 강화 설비 세트 150G
```

따라서 재플레이와 높은 별 없이도 R5 필수 구매가 가능하다.

## 10. 수치 성립 검증

### 10.1 R1 — 첫 성공 안전성

- 단검 한 개의 최소 경로는 14초다.
- 첫 의뢰 인내 45초와 라운드 기한 90초 안에 충분히 들어온다.
- 한 건 점수 10이 별 1 커트라인과 같으므로 첫 납품으로 클리어가 성립한다.

### 10.2 R2 — 전량 처리 가능성

- 전체 철 수요는 단검 세 개와 철검 세 개로 총 9개다.
- 용광로 순수 작업 시간은 72초, 제작대 순수 작업 시간은 45초다.
- 120초 안에서 두 시설을 병렬 운용하면 전량 납품 가능한 여유가 있다.

### 10.3 R3 — 트리아지 강제와 회복 가능성

- 요청 전체 철 수요는 단검 7개 + 철검 7개 = 철 주괴 21개다.
- 용광로는 120초 동안 이론상 최대 15회만 작업하므로 전량 처리는 불가능하다.
- 과열 소실 한 번 뒤 사용할 수 있는 철이 줄어도 철검 중심 선택으로 120점 이상을 만들 수 있어 별 3 가능성을 남긴다.
- 철회는 감점이 아니므로 낮은 점수 의뢰를 흘려보내는 전략이 성립한다.

### 10.4 R4 — 예고 의뢰의 필요성

- 예고 의뢰를 제외한 최대 점수는 100점이다.
- 별 2 커트라인 80은 넘지만 별 3 커트라인 120에는 못 미친다.
- 예고 철검을 납품하면 최대 140점으로 별 3이 가능하다.
- 공개 후 남은 24초는 원자재부터 철검을 만드는 최소 25초보다 짧다.

### 10.5 R5 — 강화 프로젝트의 필요성

- 최종 예고 의뢰를 제외한 최대 점수는 130점이다.
- 별 2 커트라인 110은 넘지만 별 3 커트라인 170에는 못 미친다.
- 최종 +1 철검을 납품하면 최대 200점으로 별 3이 가능하다.
- 공개 후 남은 35초는 +1 철검 전체 경로 46초보다 짧지만, 철검과 강화석 비축 후 필요한 12초보다 길다.
- 중반 +1 단검의 인내 70초는 전체 경로 38초를 충분히 감당한다.

## 11. 조정 순서

플레이테스트에서 문제가 생기면 다음 순서로만 조정한다.

1. UX 안내가 규칙을 정확히 전달했는지 확인한다.
2. 요청 공개 시점과 인내를 조정한다.
3. 라운드 커트라인을 조정한다.
4. 작업 시간과 과열 유예를 조정한다.
5. 요청 점수와 경제 값을 마지막에 조정한다.

콘텐츠 수나 새 시스템을 추가해 밸런스 문제를 덮지 않는다. 조정으로 R4·R5의 “예고 의뢰 없이는 별 3 불가, 별 2 가능” 관계와 R5 무재플레이 구매 보장을 깨뜨려서는 안 된다.

| `TECH-DATA-001` | 생성 데이터 값 | 실행 JSON 스키마·투영 검증 |
| `QA-MVP-001` | 검증할 기대 관계 | 플레이테스트와 자동 검증 결과 |

## 13. 미결정 사항

| 질문 | 현재 기준선 | 판정 지표 |
|---|---|---|
| 과열 유예 8초가 경고를 보고 대응하기에 충분한가? | 유지 | 첫 R3 과열률, 경고 후 회수 성공률 |
| 인벤토리 4칸이 R4·R5 비축에 충분하면서 선택을 만드는가? | 유지 | 피크 점유율, 만석 거절, 자발적 파기 |
| R3 별 3 커트라인 120이 한 번의 소실을 용서하는가? | 유지 | 의도적 소실 테스트와 실제 상위 점수 |
| R5 180초가 강화 학습과 최종 비축을 함께 수용하는가? | 유지 | 첫 +1 완료율, 최종 의뢰 완수율 |
| 실패 보상 0G가 재도전을 위축시키는가? | 유지 | 실패 후 즉시 재도전 비율과 인터뷰 |

## 14. 검증 방법

- 문서 표를 데이터로 옮긴 뒤 ID 참조 무결성을 자동 검사한다.
- R1~R5 전량을 결정론적 시뮬레이션으로 반복해 같은 명령 로그의 결과가 같은지 확인한다.
- §10의 산술 관계를 CI 회귀식으로 고정한다.
- 자동 봇 수치는 가능성 검증에만 사용하고 재미 판정은 관찰 플레이테스트로 한다.
- 값이 바뀌면 버전과 변경 이유, 영향을 받는 §10 부등식을 함께 기록한다.

## 15. 완료 기준

- 모든 MVP 조정값이 이 문서 한 곳에 존재한다.
- R1~R5의 모든 이벤트가 공개 시점·점수·인내·커트라인을 가진다.
- R3 트리아지, R4 비축, R5 강화 비축의 수치적 필요성이 계산으로 성립한다.
- 별 1 진행만으로 강화 설비 구매와 계약 도달이 가능하다.
- `TECH-SIM-001`과 `TECH-DATA-001`이 추측 없이 판정·데이터를 구현할 수 있고 `QA-MVP-001`이 회귀식을 만들 수 있다.
- MVP 밖의 값이나 상업 수치가 없다.
